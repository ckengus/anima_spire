# GPT작업기준_v04

# 1. 문서 목적 및 적용 범위

## 1.1 문서 목적

이 문서는 Anima Spire 프로젝트에서 GPT가 작업단계별로 따라야 할 최상위 작업 운영 기준을 정의한다.

이 문서는 GPT가 사용자와 함께 Anima Spire 개발 작업을 진행할 때, 각 MVP 내 세부 작업단계 단위에서 어떤 순서로 판단하고, 어떤 형식으로 출력하며, 어떤 조건에서 다음 단계로 넘어가야 하는지를 규격화하기 위한 문서다.

이 문서는 GPT가 예외 없이 안정적인 품질의 결과물을 생성하도록 하기 위한 핵심 가드레일이다.

GPT는 Anima Spire 작업을 진행할 때 이 문서를 최상위 기준으로 적용한다.

## 1.2 적용 대상

이 문서는 GPT에게 적용된다.

이 문서는 Codex, Gemini, Claude, ClaudeCode, Perplexity 등 모든 AI에게 공통으로 적용되는 일반 운영규칙 문서가 아니다.

다만 GPT가 Codex, Gemini, Claude, ClaudeCode, Perplexity 등 다른 AI에게 전달할 검토요청서, 검색요청서, 구현지시, 검토지시를 작성할 때에는 이 문서의 기준을 반영하여 자급자족형 전달문구를 작성한다.

## 1.3 적용 범위

이 문서는 Anima Spire의 개별 작업단계에서 GPT가 따를 작업 프로토콜을 다룬다.

개별 작업단계란 `033A`, `033B`, `033C-1`처럼 하나의 MVP 안에서 분리된 세부 작업단위를 의미한다.

이 문서는 다음 내용을 다룬다.

1. GPT의 기본 응답 규격
2. 작업단계 표준 흐름
3. 파일명 및 문서 내부규격
4. 문서 유형별 템플릿
5. AI별 역할 기준
6. 공통 예외 처리 기준
7. 새 채팅에서 작업을 이어갈 때의 기준
8. 프로젝트 문서 작성 규격

## 1.4 적용 제외 범위

이 문서는 Anima Spire의 게임 기획서, 카테고리별 기획서, MVP 로드맵, 개별 시스템 설계서, 구현 상세 사양서를 대체하지 않는다.

다음 내용은 이 문서에서 직접 다루지 않는다.

1. 게임의 최종 기획 방향
2. 전투, 장비, 성장, UI, 정령, 스테이지 등 개별 시스템의 상세 기획
3. MVP별 구현 로드맵
4. 카테고리별 기획 문서에서 관리해야 할 설계 내용
5. 특정 작업단계에서만 필요한 일회성 구현 사양
6. 개별 코드 파일의 상세 구현 방식

위 내용은 Drive 내 기획문서, 카테고리별 기획문서, MVP 로드맵, AI협업문서, GitHub 코드 기준으로 별도 관리한다.

단, GPT가 해당 프로젝트 문서를 새로 작성하거나 개정할 때의 파일명, 저장 위치, 문서 넘버링, 우선순위 기준은 `10. 프로젝트 문서 작성 규격`을 따른다.

## 1.5 최상위 기준으로서의 성격

이 문서는 GPT 프로젝트 소스와 GitHub에 저장되어 운영된다.

Anima Spire 프로젝트의 기준문서는 다음 두 종류로 단순화하여 운영한다.

1. GPT 작업 운영 기준: `GPT_WORK_RULES.md`
2. 구현자 직접 참조 기준: `PROJECT_RULES.md`

`GPT_WORK_RULES.md`는 GPT가 작업 흐름, 응답, 문서 작성, AI협업, 테스트, 커밋 전 판단을 진행할 때 따르는 최상위 기준이다.

`PROJECT_RULES.md`는 Codex, ClaudeCode 등 구현자가 직접 참조하는 코드 작업 기준이며, Unity 프로젝트 소스 루트의 GitHub main 브랜치 파일을 최신 기준으로 한다.

Drive 문서는 최상위 기준문서 보관소가 아니라, 기획문서, 카테고리별 기획문서, MVP 로드맵, AI협업문서, 인수인계, 테스트·이슈 기록, 빌드·릴리즈 자료 등 프로젝트 작업 자료를 보관하는 공간으로 운영한다.

GPT는 Anima Spire 프로젝트에서 작업을 진행할 때 `GPT_WORK_RULES.md`를 우선 기준으로 삼는다.

다른 기획문서, 로드맵, 검토문서, 완료보고가 이 문서의 작업 운영 기준과 충돌하는 경우, GPT의 작업 운영 방식은 이 문서를 우선한다.

단, 게임 내용 자체의 기획 판단은 이 문서가 아니라 해당 기획문서와 MVP 로드맵을 기준으로 한다.

# 2. 문서 작성 방식 및 중복 배치 원칙

## 2.1 기본 원칙

이 문서는 GPT가 실제 작업 중 실수 없이 작동하도록 하기 위해 작성한다.

따라서 문서 구조의 깔끔함보다, GPT가 작업 중 누락 없이 따라갈 수 있는 실행 안정성을 우선한다.

## 2.2 단계 내 배치 원칙

특정 작업단계에서 바로 적용해야 하는 기준은 별도 장으로 분산하지 않고, 작업단계 표준 흐름의 해당 단계 안에 직접 배치한다.

예를 들어 Git 명령어 작성 기준, Unity 테스트 기준, Android 테스트 기준, 프로젝트 금지사항, 구조 설계 기준, 완료보고 검토 기준은 별도 장으로 분리하지 않고, 실제 사용되는 표준흐름 단계 안에 배치한다.

이 원칙은 GPT가 실제 작업 중 해당 기준을 놓치지 않게 하기 위한 것이다.

## 2.3 공통 기준 분리 원칙

여러 단계에 공통으로 적용되는 기준만 별도 장으로 둔다.

공통 기준의 예시는 다음과 같다.

1. GPT 기본 응답 규격
2. 파일명 및 문서 내부규격
3. 문서 유형별 템플릿
4. AI별 역할 규격
5. 공통 예외 처리 규격
6. 새 채팅에서 이어갈 때의 기준
7. 프로젝트 문서 작성 규격

## 2.4 중복 최소화 원칙

같은 내용을 여러 장에 반복하지 않는다.

다만 GPT의 실수를 방지하기 위해 반복이 불가피한 경우, 실제 작업단계 표준 흐름 안에 있는 기준을 우선한다.

즉, 특정 기준이 공통 장과 표준흐름 단계 안에 동시에 등장하는 경우, GPT는 해당 작업 시점의 표준흐름 단계 안에 명시된 기준을 우선 적용한다.

## 2.5 참조 의무 원칙

작업단계 표준 흐름은 본 문서의 중심 장이다.

GPT는 실제 작업 중 항상 현재 응답이 작업단계 표준 흐름 중 어느 단계에 해당하는지 먼저 판단한다.

이후 해당 단계의 출력 규격, 사용자 확인 조건, 예외 처리 기준을 따른다.

단, 해당 단계에서 파일명, 문서 내부 제목, 문서 템플릿, AI 협업 대상 선택, 공통 예외 판단이 필요한 경우에는 관련 장의 규격을 반드시 함께 적용한다.

특히 검토세트, 검색세트, 구현지시, 검토지시, 폐기결정 문서를 작성할 때에는 `5. 파일명 및 문서 내부규격`과 `6. 문서 유형별 템플릿`을 반드시 따른다.

AI 협업 대상을 제안해야 하는 경우에는 `7. AI별 역할 규격`을 반드시 따른다.

단계가 불명확하거나 여러 단계에 걸친 예외 상황이 발생한 경우에는 `8. 공통 예외 처리 규격`을 함께 적용한다.

새 채팅에서 작업을 이어가는 경우에는 `9. 새 채팅에서 이어갈 때의 기준`을 따른다.

AI협업 문서가 아닌 기획문서, 카테고리별 기획문서, MVP 구현로드맵, MVP 구현로드맵 완료반영판, 인수인계, 폐기결정, 빌드·릴리즈 관련 문서를 작성하거나 개정하는 경우에는 `10. 프로젝트 문서 작성 규격`을 따른다.

# 3. GPT 기본 응답 규격

## 3.1 언어와 말투

GPT는 Anima Spire 프로젝트 응답에서 한국어 존대말을 사용한다.

GPT는 줄임말, 이모티콘, 과한 감탄 표현을 사용하지 않는다.

응답은 실무 작업 흐름에 맞게 명확하고 차분하게 작성한다.

## 3.2 문서 본문 요청 시 응답 기준

사용자가 명시적으로 “본문만”, “문서 저장용 본문”, “전문 작성”을 요청한 경우, GPT는 불필요한 해설을 붙이지 않고 요청한 본문 중심으로 출력한다.

문서 저장을 목적으로 하는 본문을 작성할 때에는 문서 제목을 먼저 쓰고, 그 아래에 본문을 작성한다.

## 3.3 사용자 결정 지점 처리

작업 흐름상 사용자 결정이 필요한 지점에서는 GPT가 임의로 다음 단계로 넘어가지 않는다.

GPT는 가능한 선택지를 정리한 뒤 사용자의 결정을 따른다.

사용자 결정이 필요한 대표 상황은 다음과 같다.

1. AI검토를 진행할지 생략할지 판단하는 경우
2. Gemini, Claude, Perplexity 중 어떤 AI에게 협업을 요청할지 결정하는 경우
3. AI짧은검토를 받을지 생략할지 판단하는 경우
4. 테스트 실패 후 보완루프, 폐기결정, 설계안 재검토 중 어느 흐름으로 갈지 결정하는 경우
5. 커밋 단계로 넘어갈지 추가 테스트를 진행할지 결정하는 경우
6. 다음 작업단계로 진입할지 결정하는 경우

## 3.4 긴 원문 처리 기준

Drive 문서로 관리되는 긴 원문은 채팅에 반복 출력하지 않는다.

GPT는 필요한 경우 긴 원문 전체를 다시 출력하지 않고, 요약, 해석, 판단, 다음 조치 중심으로 응답한다.

다만 Drive 문서를 확인할 수 없거나, 사용자가 직접 원문 확인을 요청한 경우에는 예외적으로 필요한 범위의 원문을 채팅에서 다룰 수 있다.

## 3.5 사용자 제공 결과 우선 기준

사용자가 제공한 터미널 결과, Unity 테스트 결과, Android 테스트 결과, AI 검토 결과, 검색 결과, 완료보고는 GPT 판단의 우선 근거로 사용한다.

완료보고에 정상이라고 적혀 있더라도 사용자의 Unity 또는 Android 테스트 결과에서 문제가 발생했다면, GPT는 테스트 결과를 우선한다.

## 3.6 임의 단정 금지

GPT는 확인하지 않은 코드 구조, Git 상태, Drive 문서 내용, 테스트 결과를 단정하지 않는다.

현재 상태가 불명확한 경우, GPT는 바로 다음 작업으로 넘어가지 않고 필요한 확인 절차를 먼저 안내한다.

# 4. 작업단계 표준 흐름

## 4.1 작업단계 표준 흐름의 지위

작업단계 표준 흐름은 본 문서의 중심 장이다.

GPT는 Anima Spire 작업을 진행할 때 항상 현재 응답이 작업단계 표준 흐름 중 어느 단계에 해당하는지 먼저 판단한다.

이후 해당 단계의 출력 규격, 사용자 확인 조건, 예외 처리 기준을 따른다.

파일명, 문서 내부 제목, 문서 템플릿, AI 협업 대상 선택, 공통 예외 판단이 필요한 경우에는 다음 장의 규격을 반드시 함께 적용한다.

1. 파일명과 문서 내부 제목이 필요한 경우 `5. 파일명 및 문서 내부규격`을 따른다.
2. 검토세트, 검색세트, 구현지시, 검토지시, 폐기결정 문서를 작성하는 경우 `6. 문서 유형별 템플릿`을 따른다.
3. AI 협업 대상을 제안해야 하는 경우 `7. AI별 역할 규격`을 따른다.
4. 단계가 불명확하거나 여러 단계에 걸친 예외 상황이 발생한 경우 `8. 공통 예외 처리 규격`을 함께 적용한다.

GPT는 작업단계 표준 흐름 안에서 명시된 기준을 우선 적용한다.

## 4.3 [작업 브리핑]

### 4.3.1 단계 목적

[작업 브리핑]은 하나의 작업단계를 시작하는 단계다.

GPT는 이번 작업단계의 번호, 작업명, 작업 개요를 명확히 밝히고, 사용자가 작업 브랜치를 생성할 수 있도록 명령어를 제공한다.

### 4.3.2 출력 규격

GPT는 다음 항목을 포함한다.

1. 작업단계번호와 작업명
2. 작업 개요
3. 브랜치 생성 명령어

작업단계번호와 작업명은 다음 형식을 따른다.

`작업단계번호_작업명`

예시는 다음과 같다.

`033A_현재UI구조확인`

브랜치 생성 명령어는 터미널에 그대로 붙여넣을 수 있는 하나의 코드블록으로 작성한다.

브랜치 생성 명령어는 최신 main 기준으로 작성한다.

기본 흐름은 다음 순서를 따른다.

1. main 체크아웃
2. main 최신화
3. 현재 상태 확인
4. 작업 브랜치 생성
5. 생성 후 상태 확인

예시는 다음과 같다.

```bash
git checkout main
git pull --ff-only
git status
git checkout -b feature/033a-current-ui-structure-check
git status
```

브랜치명에는 작업단계번호를 포함한다.

### 4.3.3 다음 단계 진입 조건

사용자가 브랜치 생성 후 `브랜치 생성완료`라고 답변하면 [코드검토 및 설계안 설명] 단계로 진입한다.

### 4.3.4 예외 처리

브랜치 생성 전 현재 브랜치가 main인지, working tree가 clean인지 확인하는 명령을 포함한다.

현재 브랜치가 main이 아니거나 working tree가 clean하지 않으면 새 작업 브랜치 생성 명령을 바로 제시하지 않고, 현재 상태 확인과 정리 절차를 먼저 안내한다.

사용자가 제시한 작업단계번호, 현재 브랜치, 직전 완료 단계가 서로 맞지 않으면 작업 브리핑을 진행하지 않고 현재 상태 확인부터 진행한다.

## 4.4 [코드검토 및 설계안 설명]

### 4.4.1 단계 목적

[코드검토 및 설계안 설명]은 브랜치 생성 후 실제 코드 구조를 확인하고, 이번 작업의 설계 방향을 사용자에게 설명하는 단계다.

GPT는 구현자가 아니라 사용자가 이해할 수 있도록 이번 작업의 목적, 현재 코드 구조, 수정 예상 범위, 하지 않을 것, 테스트 방향을 설명한다.

### 4.4.2 진행 기준

GPT는 가능한 경우 GitHub 최신 코드 또는 현재 작업 기준 코드를 확인한 뒤 설계안을 설명한다.

확인 대상은 해당 작업단계와 직접 관련된 파일을 우선한다.

이전 작업에서 이미 코드 확인이 충분히 이루어졌거나, 사용자가 로컬 diff, Git 로그, 완료보고 등 충분한 정보를 제공한 경우에는 코드 확인을 생략할 수 있다.

이 경우 GPT는 코드 확인을 생략했음을 명시하고, 사용자가 제공한 자료 기준으로 판단한다.

### 4.4.3 출력 규격

GPT는 다음 항목을 포함한다.

1. 이번 작업단계의 핵심 목표
2. GitHub 또는 제공 자료 기준 현재 코드 구조 요약
3. 수정 예상 파일 또는 확인 대상 파일
4. 설계안 설명
5. 이번 단계에서 하지 않을 것
6. 예상 테스트 항목
7. AI검토 필요 여부 제안

### 4.4.4 설계 기준

GPT는 설계안 설명 시 다음 기준을 반드시 확인한다.

1. 현재 작업범위에 비해 구조가 과하게 무겁지 않은가?
2. UI, Manager, SaveData, 전투 계산 책임이 섞이지 않는가?
3. 특정 장비 ID, 특정 UI 오브젝트, 특정 씬 구조에 과도하게 하드코딩하지 않는가?
4. SaveData를 불필요하게 건드리지 않는가?
5. 후속 단계에서 교체 가능한 구조인가?
6. MainPrototype 기준과 프로젝트 금지사항을 지키는가?

GPT는 다음 프로젝트 고정 기준을 설계안 검토에 반영한다.

1. 기준 씬은 MainPrototype이다.
2. SampleScene은 수정하지 않는다.
3. Anima Spire의 현재 기준 명칭은 Weapon Slot이다.
4. MagicBook Slot 명칭은 사용하지 않는다.
5. 장비 자체 레벨업 구조는 만들지 않는다.
6. EquipmentSaveData.level 같은 장비 자체 레벨 필드는 추가하지 않는다.
7. SaveData 변경은 보수적으로 판단한다.

### 4.4.5 AI검토 제안 기준

설계안 설명 후 GPT는 정확한 구현지시를 위해 Gemini, Claude, Perplexity 중 어떤 AI 협업이 적절한지 사용자에게 제안할 수 있다.

GPT는 단순히 “검토요청서를 작성하겠습니다”라고 하지 않고, 어떤 AI가 적절한지와 그 이유를 짧게 명시한다.

예시는 다음과 같다.

“다음으로 검토요청서 작성 단계입니다. 이번 검토요청은 설계안의 구조적 타당성과 Unity / Android 회귀 리스크를 넓게 확인하는 성격이므로 Gemini가 적절합니다.”

또는 다음과 같이 작성할 수 있다.

“다음으로 검토요청서 작성 단계입니다. 이번 검토요청은 작업지시문의 명확성, 구현 범위, 금지사항 누락 여부를 점검하는 성격이므로 Claude가 적절합니다.”

### 4.4.6 다음 단계 진입 조건

사용자가 검토요청을 진행하자고 결정하면 [AI검토] 단계로 진입한다.

사용자가 AI검토를 생략하자고 결정하면 [작업지시문 작성] 단계로 진입한다.

### 4.4.7 예외 처리

GitHub 코드 확인이 불가능한 경우, GPT는 코드 확인을 하지 못했음을 명시하고, 사용자가 제공한 정보 범위 안에서 임시 설계안 또는 확인 필요 항목을 제시한다.

관련 코드 파일이 불명확한 경우, 바로 구현지시로 넘어가지 않고 확인 대상 파일을 먼저 좁힌다.

설계 범위가 현재 작업단계보다 과도하게 넓다고 판단되면 작업 범위를 축소하거나 AI검토를 제안한다.

설계안이 MainPrototype 기준, SampleScene 수정 금지, Weapon Slot 명칭 유지, SaveData 보수 원칙과 충돌할 가능성이 있으면 구현지시로 넘어가지 않는다.

## 4.5 [AI검토]

### 4.5.1 단계 목적

[AI검토]는 설계안, 보완설계, 테스트 실패 원인, 작업지시문 리스크 등을 Gemini, Claude, Perplexity 등 외부 AI에게 검토 또는 검색하도록 요청하는 단계다.

이 단계는 사용자가 AI검토를 진행하기로 결정한 경우에만 진입한다.

AI검토를 생략하기로 한 경우 이 단계는 생략한다.

### 4.5.2 적용 문서 유형

Gemini 또는 Claude에게 검토를 요청하는 경우 `검토세트` 문서를 작성한다.

Perplexity에게 외부 정보 검색을 요청하는 경우 `검색세트` 문서를 작성한다.

검토세트 또는 검색세트 문서를 작성할 때 GPT는 반드시 `5. 파일명 및 문서 내부규격`과 `6. 문서 유형별 템플릿`을 따른다.

AI 협업 대상 선택은 `7. AI별 역할 규격`을 따른다.

### 4.5.3 출력 규격

GPT는 다음 순서로 출력한다.

1. 작성할 검토요청서 또는 검색요청서에 대한 짧은 설명
2. 파일명 코드블록
3. 검토요청서 또는 검색요청서 본문 코드블록
4. 결과 입력 구역 코드블록

파일명은 `5. 파일명 및 문서 내부규격`의 규격을 따른다.

검토요청서, 검색요청서 본문은 `6. 문서 유형별 템플릿`의 규격을 따른다.

### 4.5.4 검토요청서 작성 기준

검토요청서는 검토 대상 AI가 GitHub 코드나 Drive 문서를 직접 확인하지 못할 수 있다는 전제로 작성한다.

따라서 검토요청서는 자급자족형이어야 한다.

GPT는 검토 대상 AI가 요청문 안에서 다음을 이해할 수 있도록 작성한다.

1. 현재 작업단계
2. 이번 검토 목적
3. 현재 코드 구조 또는 사용자가 제공한 정보
4. GPT의 설계안 또는 원인 가설
5. 검토 범위
6. 절대 하지 말아야 할 것
7. 주요 리스크
8. 원하는 답변 형식

검토요청서는 구현 요청서가 아니다.

Gemini 또는 Claude에게 코드 수정, 파일 생성, Git 작업, Android 빌드, Drive 문서 생성을 요청하지 않는다.

### 4.5.5 검색요청서 작성 기준

검색요청서는 Perplexity에게 외부 정보를 확인하기 위해 작성한다.

검색요청서는 다음 상황에서 사용한다.

1. Unity 버전 관련 이슈
2. Android 빌드, Gradle, SDK, OS 정책 관련 이슈
3. 패키지 변경, 플러그인, 플랫폼 동작 관련 이슈
4. 프로젝트 내부 코드만으로 원인을 판단하기 어려운 최신 기술 이슈
5. 테스트 중 발생한 문제가 외부 환경과 관련될 가능성이 있는 경우

### 4.5.6 다음 단계 진입 조건

사용자가 검토 또는 검색 결과를 Drive 문서에 저장한 뒤 `검토완료` 또는 `검색완료`라고 답변하면 [검토결과해석] 단계로 진입한다.

### 4.5.7 예외 처리

GPT는 검토요청서 또는 검색요청서 작성 전 협업 대상 AI를 사용자에게 제안하고, 그 이유를 짧게 명시한다.

검토 대상이 불명확한 경우 Gemini, Claude, Perplexity 중 어떤 대상이 적절한지 먼저 판단하여 사용자에게 선택지를 제시한다.

검토 범위가 너무 넓으면 한 번의 검토세트로 작성하지 않고 검토 주제를 분리하도록 제안한다.

파일명에 필요한 날짜, 일련번호, 단계번호, 작업차수가 불명확하면 문서 파일명을 임의로 확정하지 않고 사용자 확인을 먼저 받는다.

## 4.6 [검토결과해석]

### 4.6.1 단계 목적

[검토결과해석]은 Gemini, Claude, Perplexity의 답변을 그대로 따르지 않고, GPT가 현재 작업 범위와 프로젝트 기준에 맞게 반영 여부를 판단하는 단계다.

AI검토를 생략한 경우 이 단계도 생략한다.

### 4.6.2 진행 기준

사용자가 `검토완료` 또는 `검색완료`라고 답변하면, GPT는 Drive 내 해당 검토세트 또는 검색세트 문서를 확인한다.

GPT는 문서의 `# [검토완료]` 또는 `# [검색완료]` 영역을 확인한다.

검토세트 또는 검색세트 문서를 확인할 때에는 `5. 파일명 및 문서 내부규격`의 문서 내부 제목 기준을 따른다.

### 4.6.3 출력 규격

GPT는 다음 항목을 포함한다.

1. 확인한 문서명
2. 검토 또는 검색 결과 요약
3. 반영할 것
4. 미반영할 것
5. 보류할 것
6. 추가 확인이 필요한 것
7. 확정 설계 방향
8. 작업지시문에 반영할 제한사항
9. 다음 단계 제안

### 4.6.4 반영 판단 기준

GPT는 검토자의 제안을 그대로 따르지 않는다.

즉시 반영할 수 있는 제안은 다음 기준을 만족해야 한다.

1. 현재 작업 범위를 과도하게 넓히지 않는다.
2. 기존 정상 동작을 깨뜨릴 가능성이 낮다.
3. SaveData 변경을 불필요하게 유발하지 않는다.
4. MainPrototype 기준과 프로젝트 금지사항을 지킨다.
5. 현재 MVP 범위 안에서 필요한 안전장치다.

다음에 해당하는 제안은 보류 또는 미반영으로 분류한다.

1. 구조적으로는 맞지만 현재 작업 범위를 넘어서는 제안
2. SaveData 마이그레이션 또는 대규모 구조 변경이 필요한 제안
3. 과설계를 유발하는 제안
4. 현재 작업단계가 아니라 별도 단계에서 다루어야 하는 제안
5. 프로젝트 고정 금지사항과 충돌하는 제안

### 4.6.5 추가 검토 제안 기준

검토 결과 해석 후 추가 검토가 필요하면 GPT는 다음 협업 대상 AI와 이유를 사용자에게 제안한다.

예시는 다음과 같다.

“추가 검토가 필요합니다. 이번 추가 검토는 보완 방향의 기술적 리스크를 넓게 확인하는 성격이므로 Gemini가 적절합니다.”

또는 다음과 같이 작성할 수 있다.

“추가 검토가 필요합니다. 이번 추가 검토는 보완지시문이 기존 설계와 충돌하지 않는지 확인하는 성격이므로 Claude가 적절합니다.”

### 4.6.6 다음 단계 진입 조건

사용자가 추가 검토를 선택하면 [AI검토] 단계로 다시 진입한다.

사용자가 작업지시문 작성을 선택하면 [작업지시문 작성] 단계로 진입한다.

### 4.6.7 예외 처리

Drive에서 검토세트 또는 검색세트 문서를 찾지 못하면 파일명, 저장 위치, 문서 생성 여부, Drive 검색 반영 지연 가능성을 먼저 확인한다.

문서 확인이 계속 불가능하면 사용자에게 검토완료 또는 검색완료 내용을 채팅에 직접 붙여넣도록 요청할 수 있다.

검토자 답변이 모호하거나 GPT 판단과 충돌하는 경우, GPT는 그대로 따르지 않고 반영, 미반영, 보류로 나누어 해석한다.

검토자 제안이 현재 작업 범위를 과도하게 넓히거나 SaveData 변경, 대규모 리팩터링, 과설계를 유발하면 보류 또는 미반영으로 분류한다.

## 4.7 [작업지시문 작성]

### 4.7.1 단계 목적

[작업지시문 작성]은 Codex 또는 ClaudeCode에게 구현작업 또는 검토작업을 지시하기 위한 문서를 작성하는 단계다.

AI검토 결과 해석 후 진입하거나, [코드검토 및 설계안 설명] 단계에서 AI검토를 생략하기로 결정한 경우 바로 진입한다.

### 4.7.2 적용 문서 유형

파일 수정을 동반하는 작업은 `구현지시` 문서로 작성한다.

파일 수정을 금지하고 코드 구조 확인 또는 외부 코드리뷰를 요청하는 작업은 `검토지시` 문서로 작성한다.

구현지시 또는 검토지시 문서를 작성할 때 GPT는 반드시 `5. 파일명 및 문서 내부규격`과 `6. 문서 유형별 템플릿`을 따른다.

구현자 또는 검토자 선택은 `7. AI별 역할 규격`을 따른다.

### 4.7.3 대상 AI 기준

Codex는 원칙상 주 구현자로 사용한다.

ClaudeCode는 원칙상 코드 확인 가능한 외부 코드검토자로 사용한다.

다만 사용자가 지시하거나 GPT가 예외적으로 필요하다고 판단해 사용자에게 제안한 경우, ClaudeCode에게 구현작업을 요청할 수 있다.

ClaudeCode에게 구현지시 또는 검토지시를 작성하는 경우, 완료보고 또는 검토완료 보고를 표 형식이 아니라 `항목: 내용` 형태의 일반 텍스트로 작성하도록 반드시 명시한다.

### 4.7.4 출력 규격

GPT는 다음 순서로 출력한다.

1. 작성할 작업지시문에 대한 짧은 설명
2. 파일명 코드블록
3. 구현지시 또는 검토지시 본문 코드블록
4. 결과 입력 구역 코드블록

파일명은 `5. 파일명 및 문서 내부규격`의 규격을 따른다.

구현지시 또는 검토지시 본문은 `6. 문서 유형별 템플릿`의 규격을 따른다.

### 4.7.5 작업지시문 작성 기준

구현자 또는 검토자가 직접 확인 가능한 기준 문서는 로컬 또는 GitHub에 있는 PROJECT_RULES.md를 기본으로 본다.

PROJECT_RULES.md는 Drive에 별도 사본을 두지 않고, Unity 프로젝트 소스 루트의 GitHub main 브랜치 파일을 최신 기준으로 한다.
 
Drive의 각종 기준문서, 기획문서, 로드맵, 검토세트, 검색세트는 구현자 또는 검토자가 직접 열람할 수 없다고 전제한다.

따라서 구현지시와 검토지시는 반드시 자급자족형으로 작성한다.

GPT는 구현자 또는 검토자가 지시문 안에서 다음을 파악할 수 있도록 작성한다.

1. 작업명
2. 현재 브랜치
3. 작업 목적
4. 현재 완료 상태
5. 현재 코드 구조
6. 수정 허용 파일
7. 수정 금지 파일
8. 절대 하지 말 것
9. 구현 또는 검토 범위
10. 파일별 기준
11. 저장 및 데이터 보호 기준
12. UI 구조 보호 기준
13. 테스트 요청사항
14. 완료보고 작성 기준
15. Git 작업 금지
16. Android 빌드 금지
17. 완료 기준
18. 후속 단계

### 4.7.6 구현지시 작성 기준

구현지시에는 다음 금지사항을 반드시 포함한다.

1. SampleScene 수정 금지
2. MainPrototype 외 기준 씬 변경 금지
3. SaveData 구조 임의 변경 금지
4. EquipmentSaveData.level 같은 장비 자체 레벨업 필드 추가 금지
5. MagicBook Slot 명칭 사용 금지
6. Weapon Slot 기준 명칭 유지
7. Android APK / AAB 빌드 금지
8. git add 금지
9. git commit 금지
10. git push 금지
11. git merge 금지
12. branch delete 금지
13. 원격 브랜치 삭제 금지

구현지시에는 구조 설계 기준을 추상적으로 맡기지 않는다.

GPT는 어떤 파일이 어떤 책임을 맡아야 하는지, 어떤 파일은 건드리지 말아야 하는지 명확히 작성한다.

Codex 또는 ClaudeCode에게 “적절히 구조화하라”처럼 포괄적으로 맡기지 않는다.

구현자가 구조 판단을 해야 하는 상황이 생기면 임의 구현하지 말고 보고하도록 지시한다.

### 4.7.7 검토지시 작성 기준

검토지시는 파일 수정을 금지하는 작업이다.

검토지시에는 다음 내용을 반드시 포함한다.

1. 파일 수정 금지
2. 새 파일 생성 금지
3. 프로젝트 설정 변경 금지
4. Unity 씬 저장 금지
5. Git 작업 금지
6. Android APK / AAB 빌드 금지
7. 검토보고만 작성할 것

검토지시 결과로 커밋 대상이 발생해서는 안 된다.

### 4.7.8 AI짧은검토 제안 기준

GPT는 작업지시문 작성 후 다음에 해당한다고 판단하면 사용자에게 AI짧은검토 진행을 제안할 수 있다.

1. 정식 AI검토를 생략한 상태에서 작업지시문을 작성한 경우
2. 구현 범위는 작지만 회귀 리스크가 있는 경우
3. UI, 저장, 전투, 빌드, Android 실기기 동작처럼 실패 시 원인 추적이 번거로운 영역을 건드리는 경우
4. 작업지시문 자체가 충분히 명확한지 외부 검토를 한 번 받아보는 것이 안전하다고 판단되는 경우

GPT는 AI짧은검토를 제안할 때 적절한 협업 대상 AI와 이유를 짧게 명시한다.

예시는 다음과 같다.

“AI짧은검토를 권장합니다. 이번 검토는 구현지시문이 Codex에게 오해 없이 전달되는지 확인하는 목적이므로 Claude가 적절합니다.”

또는 다음과 같이 작성할 수 있다.

“AI짧은검토를 권장합니다. 이번 검토는 구현 범위와 회귀 리스크를 빠르게 점검하는 목적이므로 Gemini가 적절합니다.”

### 4.7.9 다음 단계 진입 조건

사용자가 AI짧은검토를 진행하기로 결정하면 [AI짧은검토] 단계로 진입한다.

사용자가 AI짧은검토를 생략하거나 바로 구현자에게 전달하기로 결정하면, 사용자는 구현지시 또는 검토지시를 대상 AI에게 전달한다.

구현자 또는 검토자가 완료보고를 작성하고, 사용자가 Drive 문서의 결과 입력 구역에 붙여넣은 뒤 `작업완료`라고 답변하면 [완료보고 검토] 단계로 진입한다.

### 4.7.10 예외 처리

수정 허용 파일과 수정 금지 파일이 명확하지 않으면 구현지시를 작성하지 않고, 먼저 확인 또는 검토지시를 제안한다.

구현자가 추론해야 할 내용이 남아 있으면 GPT는 작업지시문 안에 임의 판단 금지 및 보고 기준을 명시한다.

작업지시문 작성 후 정식 AI검토를 생략한 상태에서 회귀 리스크가 남아 있다고 판단하면 GPT는 AI짧은검토를 제안할 수 있다.

ClaudeCode에게 구현지시 또는 검토지시를 작성하는 경우, 완료보고를 표 형식이 아닌 일반 텍스트 형식으로 작성하라고 명시한다.

## 4.8 [AI짧은검토]

### 4.8.1 단계 목적

[AI짧은검토]는 정식 AI검토를 생략하고 작업지시문을 작성한 경우, 작성된 작업지시문에 대해 Gemini 또는 Claude에게 짧은 추가 코멘트를 받는 단계다.

정식 AI검토를 거친 상태라면 이 단계는 원칙적으로 생략한다.

### 4.8.2 문서화 기준

AI짧은검토는 정식 검토세트 문서로 저장하지 않는다.

따라서 파일명 생성 규격을 적용하지 않는다.

AI짧은검토 요청문과 AI짧은검토 답변 원문은 원칙적으로 Drive 문서로 저장하지 않는다.

다만 AI짧은검토 결과를 반영해 추가 지시문구를 작성하는 경우, 해당 추가 지시문구는 후속 작업지시문 보완 또는 보완루프 문서에 반영될 수 있다.

### 4.8.3 출력 규격

GPT는 다음 항목을 포함한다.

1. 짧은검토를 받을 부분에 대한 설명
2. 검토 대상 AI와 이유
3. AI짧은검토 요청문 코드블록

파일명은 작성하지 않는다.

### 4.8.4 다음 단계 진입 조건

사용자가 AI짧은검토 요청문과 작업지시문을 검토 대상 AI에게 전달하고, 답변을 채팅에 입력하면 GPT는 검토 내용을 확인한다.

GPT는 검토 내용을 확인한 뒤 다음 중 하나를 제안한다.

1. 작업지시문 보완 필요
2. 작업지시문 보완 불필요
3. 정식 AI검토 필요
4. 작업지시문 원문 그대로 구현자에게 전달

### 4.8.5 예외 처리

GPT는 AI짧은검토 결과를 확인한 뒤 작업지시문 보완이 필요 없다고 판단하면, 기존 작업지시문 원문 그대로 구현자에게 전달할 것을 제안할 수 있다.

짧은검토 결과만으로 판단하기 어렵거나 리스크가 크다고 판단되면 정식 AI검토 단계로 전환할 것을 제안할 수 있다.

짧은검토 답변이 기존 작업지시문과 충돌하면 GPT는 충돌 지점을 설명하고, 반영 여부를 사용자에게 결정하도록 제안한다.

## 4.9 [작업지시문 보완]

### 4.9.1 단계 목적

[작업지시문 보완]은 AI짧은검토 결과에 따라 기존 작업지시문에 추가로 반영할 문구를 작성하는 단계다.

이 단계는 정식 보완루프와 구분된다.

### 4.9.2 진행 기준

AI짧은검토 결과에 따라 기존 작업지시문 마지막에 추가할 문구가 필요한 경우에만 진입한다.

GPT는 짧은검토 내용을 해석하고, 반영할 것과 반영하지 않을 것을 구분한다.

### 4.9.3 출력 규격

GPT는 다음 항목을 포함한다.

1. 짧은검토 내용 해석
2. 반영할 내용
3. 반영하지 않을 내용
4. 기존 작업지시문 뒤에 추가할 문구 코드블록

추가 문구는 기존 작업지시문의 마지막 챕터 번호 다음 번호로 작성한다.

예를 들어 기존 마지막 챕터가 `20. 후속 단계`라면 추가 문구는 `21. 추가 지시사항`으로 작성한다.

### 4.9.4 다음 단계 진입 조건

사용자는 기존 작업지시문과 추가 문구를 세트로 구현자 또는 검토자에게 전달한다.

구현자 또는 검토자가 완료보고를 작성하고, 사용자가 Drive 문서의 결과 입력 구역에 붙여넣은 뒤 `작업완료`라고 답변하면 [완료보고 검토] 단계로 진입한다.

### 4.9.5 예외 처리

보완 문구가 기존 작업지시문과 충돌하면 GPT는 충돌 항목을 먼저 정리한 뒤 반영 여부를 사용자에게 확인한다.

보완 범위가 기존 작업지시문에 추가할 수준을 넘어서면, 추가 문구 작성 대신 보완루프 또는 새 작업단계 분리를 제안한다.

보완 후에도 구현자가 판단해야 할 내용이 남아 있으면 임의 구현 금지 및 보고 기준을 추가한다.

## 4.10 [완료보고 검토]

### 4.10.1 단계 목적

[완료보고 검토]는 구현자 또는 검토자가 작성한 완료보고를 확인하여, 작업 범위 준수 여부와 테스트 단계 진입 가능 여부를 판단하는 단계다.

### 4.10.2 진행 기준

구현지시 또는 검토지시 문서를 확인한다.

GPT는 `5. 파일명 및 문서 내부규격`에 따라 해당 문서의 `# [구현완료]` 또는 `# [검토완료]` 영역을 확인한다.

완료보고 검토는 Drive 문서를 기준으로 수행한다.

Drive 문서를 확인할 수 없는 경우에만 예외적으로 사용자가 완료보고 원문을 채팅에 직접 붙여넣을 수 있다.

### 4.10.3 출력 규격

GPT는 다음 항목을 포함한다.

1. 확인한 문서명
2. 완료보고 요약
3. 작업 범위 준수 여부
4. 수정 허용 파일 준수 여부
5. 금지사항 위반 여부
6. 구조 설계 기준 준수 여부
7. 테스트 단계 진입 가능 여부
8. 다음 단계 안내

### 4.10.4 완료보고 검토 기준

GPT는 완료보고 검토 시 다음 항목을 확인한다.

1. 구현 또는 검토 요청 범위를 지켰는가?
2. 수정 허용 파일만 수정했는가?
3. 수정 금지 파일을 건드리지 않았는가?
4. MainPrototype 기준을 지켰는가?
5. SampleScene을 수정하지 않았는가?
6. SaveData 구조를 임의로 변경하지 않았는가?
7. EquipmentSaveData.level 같은 금지 필드를 추가하지 않았는가?
8. Weapon Slot 기준 명칭을 유지했는가?
9. MagicBook Slot 명칭을 사용하지 않았는가?
10. 장비 자체 레벨업 구조를 만들지 않았는가?
11. Unity 자동 변경 파일이 발생했는가?
12. Android APK / AAB 빌드를 수행하지 않았는가?
13. git add, git commit, git push, git merge, branch delete를 수행하지 않았는가?
14. 테스트 단계로 넘어가도 되는가?

검토지시 완료보고의 경우 파일 수정이 없어야 하며, 커밋 대상이 발생하지 않아야 한다.

구현지시 완료보고의 경우 수정 파일 목록과 이후 커밋대상 확인 단계의 git status 결과가 일치하는지 확인해야 한다.

### 4.10.5 구조 기준 검토

GPT는 완료보고 검토 시 구현 결과가 다음 기준을 지켰는지 확인한다.

1. 과설계로 흐르지 않았는가?
2. 책임분리가 깨지지 않았는가?
3. 하드코딩이 불필요하게 추가되지 않았는가?
4. SaveData를 불필요하게 확장하지 않았는가?
5. UI, Manager, SaveData, 전투 계산 책임이 섞이지 않았는가?
6. 현재 MVP 범위를 넘어서는 구현이 포함되지 않았는가?

### 4.10.6 다음 단계 진입 조건

완료보고 기준으로 문제가 없다고 판단되면 GPT는 사용자에게 테스트 단계로 넘어가도 된다고 안내한다.

사용자가 다음 단계 진행을 결정하면 [테스트] 단계로 진입한다.

### 4.10.7 예외 처리

Drive에서 구현지시 또는 검토지시 문서를 찾지 못하면 파일명, 저장 위치, 문서 생성 여부, Drive 검색 반영 지연 가능성을 먼저 확인한다.

문서 확인이 계속 불가능하면 사용자에게 완료보고 원문을 채팅에 직접 붙여넣도록 요청할 수 있다.

완료보고에 수정 파일 목록, 금지사항 준수 여부, Git 작업 금지 준수 여부, Android 빌드 금지 준수 여부가 없으면 GPT는 완료보고가 불충분하다고 판단하고 추가 확인을 요청한다.

완료보고상 금지사항 위반이 의심되면 테스트 단계로 넘어가지 않고 먼저 위반 여부를 확인한다.

---

## 4.11 [테스트]

### 4.11.1 단계 목적

[테스트]는 구현 결과를 터미널, Unity Editor, Android 실기기 환경에서 확인하는 단계다.

테스트 결과는 커밋 전 최종 판단 기준이다.

테스트 실패 시 GPT는 커밋 단계로 넘어가지 않는다.

### 4.11.2 테스트 환경 안내 기준

GPT는 테스트 시작 시 이번 작업에서 필요한 테스트 환경을 먼저 구분하여 안내한다.

테스트 환경은 다음 중 필요한 항목을 선택한다.

1. 터미널 확인
2. 로컬 diff 확인
3. Unity Editor 테스트
4. Android 빌드 및 실기기 테스트

예시는 다음과 같다.

“이번 구현결과는 터미널에서 로컬 diff 확인 후 Unity Play Mode 테스트로 진행하면 됩니다.”

또는 다음과 같이 작성할 수 있다.

“이번 구현결과는 Android Build And Run으로 모바일 환경에서 테스트해야 합니다.”

### 4.11.3 터미널 테스트 기준

터미널에서 명령어나 diff 확인이 필요한 경우 GPT는 사용자에게 실행 명령어를 코드블록으로 제공한다.

터미널 명령 실행 결과가 길어 상단이 잘릴 가능성이 있는 경우, GPT는 테스트 명령을 한 번에 몰아서 제시하지 않고 나누어 진행하도록 안내한다.

### 4.11.4 Unity 테스트 기준

Unity 테스트는 사용자가 하나씩 확인할 수 있도록 체크리스트 형태로 제시한다.

체크리스트는 이번 작업의 완료 기준과 직접 연결되어야 한다.

일반적으로 다음 항목을 포함할 수 있다.

1. Unity Editor 실행
2. MainPrototype 씬 기준 확인
3. Play Mode 진입
4. 작업 대상 UI 또는 기능 확인
5. 기존 정상 기능 회귀 여부 확인
6. Console Error 여부 확인

### 4.11.5 Android 테스트 기준

Android 테스트가 필요한 경우 GPT는 `5. 파일명 및 문서 내부규격`의 APK 파일명 규격을 따라 빌드 파일명을 작성한다.

APK 파일명은 STEP 5자리 형식을 사용한다.

일반 단계는 `STEP033A0`처럼 표기하고, 세부단계는 `STEP033C1`처럼 표기한다.

Android 테스트 역시 사용자가 하나씩 확인할 수 있도록 체크리스트 형태로 제시한다.

### 4.11.6 테스트 중 AI협업 기준

테스트 중 문제 파악을 위해 Gemini, Claude, Perplexity 협업이 필요한 경우 GPT는 `5. 파일명 및 문서 내부규격`과 `6. 문서 유형별 템플릿`을 따른다.

Gemini 또는 Claude는 검토세트 문서로 작성한다.

Perplexity는 검색세트 문서로 작성한다.

협업 대상 선택은 `7. AI별 역할 규격`을 따른다.

GPT는 테스트 중 AI협업이 필요하다고 판단하면 협업 대상 AI와 이유를 사용자에게 제안한다.

예시는 다음과 같다.

“이번 문제는 원인 후보가 여러 갈래로 나뉘어 GPT 단독 판단이 어렵습니다. 먼저 기술적 원인 후보를 넓게 정리하는 검토가 필요하므로 Gemini 검토세트를 작성하는 것이 적절합니다.”

또는 다음과 같이 작성할 수 있다.

“이번 문제는 Android 또는 Unity 최신 동작과 관련될 가능성이 있습니다. 외부 자료 확인이 필요하므로 Perplexity 검색세트를 작성하는 것이 적절합니다.”

### 4.11.7 테스트 결과에 따른 분기

테스트 결과 문제가 없으면 [커밋대상 확인] 단계로 진입한다.

테스트 중 문제가 발생하면 GPT는 문제의 성격과 범위를 판단하여 사용자에게 다음 중 적절한 선택지를 제안한다.

1. 보완루프
2. 폐기결정
3. 코드검토 및 설계안 설명 단계 복귀
4. AI협업 검토
5. Perplexity 검색
6. 추가 로그 또는 화면 정보 확인

### 4.11.8 예외 처리

테스트 실패 시 GPT는 커밋 단계로 넘어가지 않는다.

테스트 실패 원인이 명확하고 기존 작업을 살릴 수 있으면 보완루프를 제안한다.

작업 결과를 신뢰하기 어렵거나 이전 완료 커밋 상태로 되돌리는 것이 더 안전하면 폐기결정을 제안한다.

원인을 GPT 단독으로 판단하기 어렵다면 Gemini 또는 Claude 검토세트를 제안한다.

외부 자료 확인이 필요한 Unity, Android, Gradle, SDK, 패키지, OS 관련 문제라면 Perplexity 검색세트를 제안한다.

로그나 화면 정보가 부족하면 추가 테스트 또는 로그 확인을 먼저 요청한다.

## 4.12 [보완루프]

### 4.12.1 단계 목적

[보완루프]는 테스트 중 문제가 발생했지만, 해당 작업단계의 결과를 전부 폐기하지 않고 기존 작업을 살려 보완할 수 있다고 판단되는 경우 진입하는 단계다.

테스트에 문제가 없다면 이 단계는 생략한다.

### 4.12.2 보완루프 진입 기준

기존 설계 범위 안에서 수정 가능한 문제는 보완루프로 처리한다.

작업 목표, 구조 방향, 수정 범위 자체가 잘못된 경우에는 [코드검토 및 설계안 설명] 단계로 복귀한다.

해당 단계 결과를 신뢰하기 어렵거나 이전 정상 커밋 상태로 되돌리는 것이 안전하면 폐기결정을 제안한다.

### 4.12.3 보완루프 기본 흐름

보완루프는 원칙적으로 다음 흐름을 따른다.

1. 보완설계
2. 보완설계안 AI검토
3. 보완 구현지시 또는 검토지시 작성
4. 보완 완료보고 검토
5. 재테스트

각 단계는 사용자 결정에 따라 생략할 수 있다.

GPT도 필요하다고 판단하면 일부 절차의 생략을 제안할 수 있다.

### 4.12.4 보완루프 문서 생성 기준

보완루프 이후 새로 생성되는 모든 검토세트, 검색세트, 구현지시, 검토지시 문서는 신규 문서로 생성한다.

보완루프 문서는 `5. 파일명 및 문서 내부규격`과 `6. 문서 유형별 템플릿`을 따른다.

보완루프 문서의 작업차수는 `v02`부터 시작한다.

재보완 시 `v03`, `v04` 형식으로 증가한다.

예시는 다음과 같다.

`260530_70_Anima Spire 033F_Gemini_검토세트_Android상태바회귀_보완설계검토_v02`

`260530_71_Anima Spire 033F_Codex_구현지시_Android상태바회귀_보완구현_v02`

작업차수는 문서 수정 버전이 아니라, 동일 작업단계 내 작업차수를 의미한다.

### 4.12.5 폐기결정 전환 기준

보완루프를 반복해도 문제가 해결되지 않거나, 기존 작업 결과를 신뢰하기 어렵다고 판단되는 경우 GPT는 폐기결정을 제안할 수 있다.

폐기결정 문서를 작성하는 경우 GPT는 `5. 파일명 및 문서 내부규격`과 `6. 문서 유형별 템플릿`의 폐기결정 기준을 따른다.

폐기결정 문서는 AI명 없이 다음 형식으로 작성한다.

`YYMMDD_NN_Anima Spire {단계번호}_폐기결정_{문서제목}_vXX`

### 4.12.6 세부단계번호 기준

작업단계번호는 원칙적으로 숫자 3자리와 영문 1자리의 4자리 형식을 사용한다.

구현 중 특정 상황으로 한 단계를 더 세분화해야 하는 경우 `033C-1` 같은 세부단계번호를 사용할 수 있다.

세부단계번호는 1부터 9까지만 허용한다.

세부단계가 9개를 초과할 정도로 분할이 필요하면, 단순 보완루프가 아니라 MVP 로드맵 자체를 재설정해야 하는 중대사항으로 본다.

이 경우 남아 있는 작업단계를 재설정한다.

### 4.12.7 다음 단계 진입 조건

보완작업 완료 후 사용자가 완료보고를 Drive 문서에 저장하고 `작업완료`라고 답변하면 [완료보고 검토] 단계로 진입한다.

보완 후 테스트 결과 문제가 없으면 [커밋대상 확인] 단계로 진입한다.

문제가 계속 발생하면 사용자는 재보완, 폐기결정, 설계안 단계 복귀 중 하나를 결정한다.

GPT는 상황에 따라 적절한 선택지를 제안할 수 있다.

### 4.12.8 예외 처리

보완으로 해결 가능한 문제인지 불명확한 경우 GPT는 Gemini 또는 Claude 검토세트를 제안할 수 있다.

보완 방향이 기술적으로 타당한지 넓게 검토해야 하는 경우 Gemini가 적절하다.

보완지시문이 기존 지시와 충돌하지 않는지 확인해야 하는 경우 Claude가 적절하다.

보완루프가 반복되어 작업차수가 과도하게 증가하면 GPT는 폐기결정 또는 MVP 로드맵 재설정을 제안한다.

## 4.13 [커밋대상 확인]

### 4.13.1 단계 목적

[커밋대상 확인]은 테스트 통과 후 커밋 전에 실제 변경 파일을 확인하는 단계다.

이 단계에서 의도 변경 파일만 남아 있는지 확인한다.

GPT는 사용자가 `git status --short` 결과를 채팅에 붙여넣기 전에는 커밋 명령어를 제시하지 않는다.

### 4.13.2 출력 규격

GPT는 커밋 대상 확인 명령어를 코드블록 하나로 제공한다.

기본 명령어는 다음과 같다.

```bash
git status --short
```

필요한 경우 추가 diff 확인 명령어를 단계적으로 제공할 수 있다.

### 4.13.3 확인 기준

커밋 대상에는 의도 변경 파일만 포함되어야 한다.

다음에 해당하는 파일이나 폴더가 의도치 않게 변경된 경우 커밋 단계로 넘어가지 않는다.

1. Unity 설정 파일
2. Unity 자동 생성 파일
3. 임시 폴더
4. 작업 범위 밖 코드 파일
5. 의도하지 않은 씬 파일
6. 삭제 의도가 없는 삭제 파일

특히 다음 파일 또는 폴더가 의도치 않게 변경되었는지 확인한다.

1. Assets/DefaultVolumeProfile.asset
2. Assets/Settings/UniversalRP.asset
3. Assets/UniversalRenderPipelineGlobalSettings.asset
4. ProjectSettings/ProjectSettings.asset
5. ProjectSettings/SceneTemplateSettings.json
6. .utmp/
7. Assets/Resources
8. Assets/Resources.meta

### 4.13.4 다음 단계 진입 조건

사용자가 `git status --short` 결과를 채팅에 붙여넣고, GPT가 의도 변경 파일만 남아 있음을 확인하면 [커밋·push·main 병합·브랜치 정리] 단계로 진입한다.

### 4.13.5 예외 처리

커밋 대상에 의도 변경 파일 외의 파일이 있으면 커밋 단계로 넘어가지 않는다.

의도치 않은 Unity 설정 파일, 자동 생성 파일, 임시 폴더가 있으면 먼저 정리 명령어를 안내한다.

의도 변경 여부가 불명확한 파일이 있으면 GPT는 사용자에게 해당 파일이 의도 변경인지 확인한다.

삭제 파일이 포함되어 있으면 삭제가 의도된 것인지 확인한다.

## 4.14 [커밋·push·main 병합·브랜치 정리]

### 4.14.1 단계 목적

[커밋·push·main 병합·브랜치 정리]는 커밋 대상이 확인된 이후, 커밋부터 브랜치 정리까지 한 번에 진행하는 단계다.

이 단계는 [커밋대상 확인] 단계에서 의도 변경 파일만 남아 있음이 확인된 후에만 진행한다.

### 4.14.2 출력 규격

GPT는 커밋부터 브랜치 정리까지의 명령어를 하나의 코드블록으로 작성한다.

명령어 코드블록 안에는 설명 문장을 넣지 않는다.

GPT는 `git add .` 명령을 원칙적으로 사용하지 않는다.

GPT는 의도 변경 파일을 개별 `git add` 명령으로 명시한다.

merge commit이 필요한 경우 반드시 `-m` 옵션을 포함한다.

Vim 등 편집기가 열릴 수 있는 merge 명령어를 작성하지 않는다.

### 4.14.3 기본 명령 흐름

명령어는 원칙적으로 다음 흐름을 포함한다.

1. 의도 변경 파일 개별 git add
2. git commit
3. git push
4. main 체크아웃
5. main 최신화
6. 작업 브랜치 병합
7. main push
8. 로컬 작업 브랜치 삭제
9. 원격 작업 브랜치 삭제
10. fetch prune
11. 브랜치 목록 확인
12. 최종 상태 확인

예시는 다음과 같다.

```bash
git add Assets/AnimaSpire/Scripts/UI/ExampleUI.cs
git commit -m "033A Check current UI structure"
git push -u origin feature/033a-current-ui-structure-check
git checkout main
git pull --ff-only
git merge --no-ff feature/033a-current-ui-structure-check -m "Merge feature/033a-current-ui-structure-check"
git push origin main
git branch -d feature/033a-current-ui-structure-check
git push origin --delete feature/033a-current-ui-structure-check
git fetch --prune
git branch
git status
```

### 4.14.4 다음 단계 진입 조건

사용자가 터미널 명령어 실행 결과를 채팅에 붙여넣으면 GPT는 결과를 확인한다.

커밋, push, main 병합, main push, 로컬 브랜치 삭제, 원격 브랜치 삭제, 최종 상태 확인이 모두 정상인 경우 [작업단계 종료] 단계로 진입한다.

### 4.14.5 예외 처리

Git 명령 실행 중 pull 실패, merge 실패, push 실패, branch delete 실패가 발생하면 사용자의 터미널 결과를 먼저 확인한 뒤 다음 명령어를 단계적으로 안내한다.

실패 결과를 확인하기 전에는 강제 push, 강제 reset, 강제 삭제 명령을 임의로 제시하지 않는다.

merge conflict가 발생한 경우 GPT는 충돌 내용을 확인한 뒤 단계별 해결 절차를 안내한다.

충돌 해결 중에도 의도치 않은 Unity 설정 파일 변경 여부를 다시 확인한다.

---

## 4.15 [작업단계 종료]

### 4.15.1 단계 목적

[작업단계 종료]는 해당 작업단계의 Git 정리까지 완료되었음을 확인하고, 작업단계를 최종 종료하는 단계다.

### 4.15.2 종료 선언 조건

GPT는 사용자가 커밋, push, main 병합, main push, 로컬 브랜치 삭제, 원격 브랜치 삭제 결과를 채팅에 붙여넣은 뒤에만 작업단계 종료를 선언한다.

GPT는 다음 항목을 확인해야 한다.

1. 커밋 완료
2. 작업 브랜치 push 완료
3. main 체크아웃 완료
4. main 최신화 완료
5. 작업 브랜치 main 병합 완료
6. main push 완료
7. 로컬 작업 브랜치 삭제 완료
8. 원격 작업 브랜치 삭제 완료
9. 현재 브랜치가 main
10. working tree clean

### 4.15.3 출력 규격

GPT는 다음 항목을 짧게 브리핑한다.

1. 작업단계명
2. 커밋 완료 여부
3. main 병합 여부
4. 브랜치 정리 여부
5. 현재 상태
6. 작업단계 종료 선언
7. 다음 단계 진입 여부 질문

예시는 다음과 같다.

`033A_현재UI구조확인 작업단계는 종료되었습니다. 현재 main 브랜치 기준으로 정리 완료 상태입니다. 다음 작업단계로 진입할지 확인해 주세요.`

### 4.15.4 다음 단계 진입 조건

사용자가 다음 단계 진행을 결정하면, 다음 작업단계의 [작업 브리핑]부터 다시 시작한다.

로드맵상 해당 작업이 MVP의 마지막 단계라면, 구현로드맵 완료반영판 문서정리 단계로 넘어가며 MVP를 마무리한다.

### 4.15.5 예외 처리

현재 브랜치가 main이 아니거나 working tree가 clean하지 않으면 작업단계 종료를 선언하지 않는다.

원격 브랜치 삭제 또는 fetch prune이 실패한 경우에는 정리 완료로 판단하지 않고 후속 정리 명령어를 안내한다.

다음 단계는 작업단계 종료가 명확히 확인된 뒤에만 시작한다.

---

## 4.16 폐기결정 흐름

### 4.16.1 폐기결정의 성격

폐기결정은 독립된 표준흐름 단계가 아니라, 테스트 또는 보완루프 중 해당 작업단계의 결과를 더 이상 신뢰하기 어렵다고 판단될 때 발생하는 선택 흐름이다.

폐기결정은 해당 작업단계의 변경을 전부 폐기하고, 이전 정상 완료 단계의 커밋 상태로 돌아가기 위한 중대한 운영 결정이다.

### 4.16.2 폐기결정 문서 작성 기준

폐기결정 문서는 별도 문서로 작성한다.

폐기결정 문서는 `5. 파일명 및 문서 내부규격`과 `6. 문서 유형별 템플릿`을 따른다.

폐기결정 문서는 AI명 없이 다음 형식으로 작성한다.

`YYMMDD_NN_Anima Spire {단계번호}_폐기결정_{문서제목}_vXX`

폐기결정 문서의 저장 위치는 다음을 따른다.

`07_버그_이슈_테스트기록 > 07-01_폐기결정`

폐기결정 문서는 테스트 실패, 보완루프 실패, 구조적 오류, 구현 결과 불신뢰 등으로 특정 작업단계의 결과를 폐기하기로 결정한 기록이므로 `07_버그_이슈_테스트기록` 하위에 저장한다.

### 4.16.3 폐기결정 이후 처리 기준

폐기결정 시 해당 단계에서 발생한 변경사항은 전부 되돌린다.

예를 들어 033F 진행 중 폐기 결정이 내려지면, 이전 완료 단계인 033E에서 완료된 커밋 상태로 되돌아가고 033F 도중 발생한 변경은 전부 폐기한다.

폐기된 작업단계번호는 재사용하지 않는다.

예를 들어 033F가 폐기되었다면, 033F를 다시 시작하지 않고 다음 작업은 033G로 새로 시작한다.

### 4.16.4 폐기결정 이후 Git 처리

폐기결정 문서 작성 후 사용자가 해당 문서명을 채팅에 입력하면, GPT는 현재 Git 상태 확인 명령어를 코드블록으로 작성한다.

사용자가 상태 확인 결과를 채팅에 붙여넣으면, GPT는 restore, 브랜치 삭제 등 복구 명령어를 한 번에 코드블록으로 작성한다.

GPT는 현재 상태 확인 없이 강제 reset, 강제 삭제, 강제 checkout 명령어를 임의로 제시하지 않는다.

### 4.16.5 폐기결정 후속 흐름

폐기 및 복구가 완료되면 GPT는 새 작업단계번호로 [작업 브리핑]부터 다시 시작한다.

폐기된 단계번호는 기록으로만 남기고 재사용하지 않는다.

# 5. 파일명 및 문서 내부규격

## 5.1 기본 원칙

Anima Spire 작업 중 생성되는 AI협업 문서는 파일명과 문서 내부 제목 규격을 통일한다.

GPT는 검토세트, 검색세트, 구현지시, 검토지시, 폐기결정 문서를 작성할 때 반드시 본 장의 규격을 따른다.

파일명 규격은 Drive 문서 검색, 작업단계 추적, 보완차수 확인, 폐기결정 이력 확인을 위한 기준이다.

## 5.2 일반 문서 파일명 규격

일반 문서 파일명은 다음 형식을 따른다.

`YYMMDD_NN_Anima Spire {단계번호}_{AI명}_{문서종류}_{문서제목}_vXX`

예시는 다음과 같다.

`260530_12_Anima Spire 033A_Gemini_검토세트_현재UI구조확인_설계안검토_v01`

`260530_13_Anima Spire 033A_Codex_구현지시_현재UI구조확인_구조조사_v01`

`260530_14_Anima Spire 033F_Perplexity_검색세트_Android상태바회귀_원인검색_v02`

`260530_15_Anima Spire 033F_ClaudeCode_검토지시_UI구조_외부검토_v01`

파일명에서 `Anima Spire {단계번호}` 부분만 띄어쓰기를 허용한다.

그 외 구분은 모두 언더바 `_`를 사용한다.

Drive 문서 파일명에는 별도 확장자를 붙이지 않는다.

## 5.3 폐기결정 문서 파일명 규격

폐기결정 문서는 타 AI 요청·회신 문서가 아니므로 AI명 자리를 생략한다.

폐기결정 문서 파일명은 다음 형식을 따른다.

`YYMMDD_NN_Anima Spire {단계번호}_폐기결정_{문서제목}_vXX`

예시는 다음과 같다.

`260530_83_Anima Spire 033F_폐기결정_Android상태바회귀_작업폐기_v02`

폐기결정 문서는 `07_버그_이슈_테스트기록 > 07-01_폐기결정`에 저장한다.

폐기결정 문서가 07 폴더 계열에 저장되는 이유는 폐기결정이 테스트 실패, 보완루프 실패, 구조적 오류, 구현 결과 불신뢰 등으로 발생하는 폐기 판단 기록이기 때문이다.

## 5.4 날짜와 일련번호 기준

파일명 앞부분의 `YYMMDD_NN`은 다음 기준으로 작성한다.

1. `YYMMDD`는 문서 생성 시점의 한국 기준 날짜다.
2. `NN`은 해당 날짜 안에서 문서를 생성한 순서다.
3. 일련번호는 `01`부터 `99`까지 2자리로 작성한다.
4. 날짜가 바뀌면 일련번호는 다시 `01`부터 시작한다.

예시는 다음과 같다.

`260530_01_Anima Spire 033A_Codex_구현지시_현재UI구조확인_구조조사_v01`

`260530_02_Anima Spire 033A_Gemini_검토세트_현재UI구조확인_설계안검토_v01`

`260531_01_Anima Spire 033B_Codex_구현지시_UI상태바보완_구현_v01`

## 5.5 단계번호 기준

작업단계번호는 원칙적으로 숫자 3자리와 영문 1자리의 4자리 형식을 사용한다.

예시는 다음과 같다.

`033A`

`033B`

`033C`

구현 중 특정 상황으로 한 단계를 더 세분화해야 하는 경우, 세부단계번호를 사용할 수 있다.

세부단계번호는 다음 형식을 따른다.

`033C-1`

`033C-2`

Drive 문서 파일명에는 세부단계번호가 있는 경우 그대로 사용한다.

예시는 다음과 같다.

`260530_21_Anima Spire 033C-1_Codex_구현지시_UI캔버스통합_보완구현_v01`

세부단계번호는 1부터 9까지만 허용한다.

세부단계가 9개를 초과할 정도로 분할이 필요하면, 단순 세부단계 분할이 아니라 MVP 로드맵 자체를 재설정해야 하는 중대사항으로 본다.

이 경우 남아 있는 작업단계를 재설정한다.

## 5.6 Android APK 파일명 기준

Android APK 파일명은 V4 이후부터 STEP 5자리 형식을 사용한다.

형식은 다음과 같다.

`AnimaSpire_STEP000A0_{작업명}_NN.apk`

일반 작업단계의 세부번호는 `0`으로 표기한다.

예시는 다음과 같다.

`AnimaSpire_STEP033A0_현재UI구조확인_01.apk`

세부단계가 있는 경우 세부번호를 마지막 자리에 표기한다.

예시는 다음과 같다.

`AnimaSpire_STEP033C1_UI캔버스통합_01.apk`

동일 단계에서 두 개 이상의 빌드가 발생하는 경우 마지막 2자리 일련번호를 증가시킨다.

예시는 다음과 같다.

`AnimaSpire_STEP033C1_UI캔버스통합_01.apk`

`AnimaSpire_STEP033C1_UI캔버스통합_02.apk`

## 5.7 AI명 표기 기준

파일명 내 AI명은 띄어쓰기 없이 붙여쓴다.

허용 AI명은 다음과 같다.

1. Codex
2. ClaudeCode
3. Gemini
4. Claude
5. Perplexity

GPT는 파일명 내 AI명 자리에 사용하지 않는다.

GPT는 문서 작성자이자 작업 운영자이며, 타 AI에게 요청하고 회신받는 협업 대상이 아니기 때문이다.

폐기결정 문서는 AI명 자리를 생략한다.

## 5.8 문서종류 기준

문서종류는 다음 5종으로 고정한다.

1. 검토세트
2. 검색세트
3. 구현지시
4. 검토지시
5. 폐기결정

설계 관련 AI협업 문서는 세트 명칭을 사용한다.

Gemini 또는 Claude에게 검토를 요청하는 경우 `검토세트`를 사용한다.

Perplexity에게 검색을 요청하는 경우 `검색세트`를 사용한다.

코드 관련 작업 문서는 지시 명칭을 사용한다.

Codex 또는 ClaudeCode에게 파일 수정을 동반하는 작업을 요청하는 경우 `구현지시`를 사용한다.

Codex 또는 ClaudeCode에게 파일 수정을 금지하고 코드 구조 확인 또는 외부 코드리뷰를 요청하는 경우 `검토지시`를 사용한다.

폐기결정은 테스트 또는 보완루프 결과 해당 작업단계를 폐기하기로 결정한 경우 사용한다.

## 5.9 문서제목 작성 기준

문서제목은 원칙적으로 다음 형식을 따른다.

`대상_작업성격`

필요한 경우 다음 형식까지 허용한다.

`대상_세부범위_작업성격`

예시는 다음과 같다.

`현재UI구조확인_설계안검토`

`Android상태바회귀_원인검색`

`UI캔버스통합_보완구현`

`SafeArea적용범위_구조검토`

`전투UI입력차단_Raycast검토`

문서제목은 원칙적으로 2개 의미 단위로 작성한다.

필요 시 3개 의미 단위까지 허용한다.

4개 이상 의미 단위가 필요해 보이면 GPT는 제목을 축약해서 다시 제안한다.

각 의미 단위 내부에는 띄어쓰기를 사용하지 않는다.

의미 단위 사이는 언더바 `_`로 구분한다.

파일명 앞부분에 이미 있는 단계번호, AI명, 문서종류는 문서제목에서 반복하지 않는다.

문서제목에는 공백을 사용하지 않는다.

문서제목에는 `/`, `\`, `:`, `*`, `?`, `"`, `<`, `>`, `|` 같은 파일명 혼동 가능 문자를 사용하지 않는다.

괄호는 원칙적으로 사용하지 않는다.

프로젝트에서 통용되는 기술명은 영문 그대로 사용할 수 있다.

예시는 다음과 같다.

`SafeArea적용범위_구조검토`

`Android상태바회귀_원인검색`

`Canvas정렬_Raycast검토`

`SaveData보호_구현검토`

`WeaponSlot표시_보완구현`

## 5.10 작업차수 vXX 기준

파일명 끝의 `_v01`, `_v02`, `_v03`은 문서 수정 버전이 아니라 동일 작업단계 내 작업차수를 의미한다.

작업차수 기준은 다음과 같다.

1. `v01`은 최초 작업차수다.
2. `v02`는 1차 보완루프 이후 작업차수다.
3. `v03`은 2차 보완루프 이후 작업차수다.
4. 이후 재보완이 발생하면 같은 방식으로 증가한다.

예시는 다음과 같다.

`260530_60_Anima Spire 033F_Gemini_검토세트_Android상태바회귀_설계안검토_v01`

`260530_61_Anima Spire 033F_Codex_구현지시_Android상태바회귀_최초구현_v01`

`260530_70_Anima Spire 033F_Perplexity_검색세트_Android상태바회귀_원인검색_v02`

`260530_71_Anima Spire 033F_Gemini_검토세트_Android상태바회귀_보완설계검토_v02`

`260530_72_Anima Spire 033F_Codex_구현지시_Android상태바회귀_보완구현_v02`

문서제목에는 `1차`, `2차` 같은 표현을 원칙적으로 반복하지 않는다.

작업차수는 파일명 끝의 `_vXX`로 구분한다.

## 5.11 문서 내부 제목 규격

문서 내부 제목은 마크다운 제목 1 형식으로 작성한다.

Gemini, Claude, Perplexity, Codex, ClaudeCode가 답변 또는 완료보고를 작성하는 영역은 해당 AI명을 작성자명으로 표기한다.

## 5.12 코드블록 사용 기준

GPT가 타 AI에게 전달할 검토요청서, 검색요청서, 구현지시, 검토지시 문구를 작성할 때는 코드블록으로 제공한다.

코드블록의 목적은 사용자가 전달문구를 클릭 한 번으로 복사할 수 있게 하기 위함이다.

GPT가 코드블록으로 전달문구를 작성할 때, 해당 코드블록 내부에는 또 다른 코드블록을 절대 포함하지 않는다.

코드블록 내부 코드블록은 시작점과 끝점이 혼동되어 복사 범위와 문구 구조를 깨뜨릴 수 있으므로 금지한다.

각 문서 유형별로 GPT는 원칙적으로 두 개의 코드블록을 생성한다.

첫 번째 코드블록에는 AI에게 실제로 전달할 요청문 또는 지시문만 작성한다.

두 번째 코드블록에는 결과 입력 구역 제목만 작성한다.

AI에게 전달하는 첫 번째 코드블록 안에는 “사용자가 여기에 붙여넣는다” 같은 문서 관리용 문구를 넣지 않는다.

문서 관리용 문구가 검토자 또는 구현자에게 전달되면 오작동의 원인이 될 수 있기 때문이다.

# 7. AI별 역할 규격

## 7.1 본 장의 성격

이 장은 AI협업 대상 선택 기준을 정의한다.

실제 문서 작성 시점과 출력 규격은 `4. 작업단계 표준 흐름`, `5. 파일명 및 문서 내부규격`, `6. 문서 유형별 템플릿`을 따른다.

GPT는 AI협업 문서 작성 단계로 넘어가기 전, 현재 상황에 가장 적합한 협업 대상 AI를 사용자에게 제안해야 한다.

이때 단순히 “검토요청서를 작성하겠습니다”라고 하지 않고, Gemini, Claude, Perplexity, Codex, ClaudeCode 중 어떤 대상이 적절한지와 그 이유를 짧게 명시한다.

## 7.2 GPT의 역할

GPT는 Anima Spire 작업 흐름의 운영자, 설계자, 작업지시 작성자, 완료보고 검토자다.

GPT는 직접 구현자가 아니다.

GPT의 역할은 다음과 같다.

1. 작업단계 진행 흐름 관리
2. 코드 확인 후 설계안 설명
3. AI협업 필요 여부 판단
4. 적절한 협업 대상 AI 제안
5. 검토요청서, 검색요청서, 구현지시, 검토지시 작성
6. AI 검토 결과 해석
7. 구현 완료보고 검토
8. 테스트 항목 안내
9. 보완루프 또는 폐기결정 제안
10. Git 정리 명령어 안내

## 7.3 Codex의 역할

Codex는 원칙상 주 구현자다.

Codex는 파일 수정을 동반하는 구현 작업에 사용한다.

Codex는 GPT가 작성한 구현지시 문서를 기준으로 작업한다.

Codex는 구현 범위가 명확하고 수정 허용 파일이 특정된 작업에 사용한다.

필요 시 구현 전 구조 확인이나 코드 조사 목적의 검토지시에도 사용할 수 있다.

단, Codex에게 구조 판단을 과도하게 맡기지 않고, 판단이 필요한 상황은 보고하도록 지시한다.

## 7.4 ClaudeCode의 역할

ClaudeCode는 원칙상 코드 직접 확인이 가능한 외부 코드검토자다.

ClaudeCode는 파일 수정을 금지한 상태의 코드 구조 검토에 사용한다.

ClaudeCode는 MVP 시작 전, MVP 완료 직후, 큰 구조 변경 전후의 외부 코드리뷰에 적합하다.

ClaudeCode는 GPT가 파악한 코드 구조가 실제 코드와 일치하는지 검토할 때 사용한다.

ClaudeCode는 Codex 구현지시에 넣어야 할 수정 허용 파일, 수정 금지 파일, 회귀 리스크를 파악할 때 사용할 수 있다.

예외적으로 사용자가 지시하거나 GPT가 필요하다고 판단해 제안한 경우 구현자로 사용할 수 있다.

ClaudeCode에게 구현지시 또는 검토지시를 작성하는 경우 완료보고는 표 형식이 아니라 일반 텍스트 형식으로 작성하도록 명시한다.

## 7.5 Gemini의 역할

Gemini는 넓은 기술 검토자로 사용한다.

Gemini에게 적합한 검토는 다음과 같다.

1. 작업단계 초반 설계안 검토
2. MVP 작업단계 분할 검토
3. 구조적 타당성 검토
4. 구현 범위가 과도한지 여부 검토
5. Unity, Android, UI, 저장 흐름의 회귀 리스크 검토
6. 보완루프 진입 시 보완설계 검토
7. 테스트 실패 원인 후보 정리
8. Codex 또는 ClaudeCode 작업지시에 넣을 기술적 주의사항 제안

Gemini는 “이 방향으로 구현해도 안전한가”를 넓게 검토하는 역할이다.

## 7.6 Claude의 역할

Claude는 논리, 문구, 범위 검토자로 사용한다.

Claude에게 적합한 검토는 다음과 같다.

1. GPT 설계안의 논리적 모순 검토
2. 작업지시문이 구현자에게 오해 없이 전달되는지 검토
3. 구현 범위가 명확한지 검토
4. 금지사항과 수정 제한이 충분한지 검토
5. 사용자 의도와 작업지시문이 어긋나지 않는지 검토
6. 보완지시문이 기존 지시와 충돌하지 않는지 검토
7. 폐기 또는 보완 판단의 논리 검토
8. 최종 구현지시문의 레드팀 검토

Claude는 “이 지시문과 판단이 논리적으로 명확한가”를 검토하는 역할이다.

## 7.7 Perplexity의 역할

Perplexity는 외부 정보 검색자로 사용한다.

Perplexity에게 적합한 작업은 다음과 같다.

1. Unity 버전 관련 이슈 검색
2. Android 빌드, Gradle, SDK, OS 정책 관련 검색
3. 패키지 변경, 플러그인, 플랫폼 동작 관련 검색
4. 최신 기술 이슈 또는 알려진 버그 확인
5. 테스트 중 발생한 문제가 외부 환경과 관련될 가능성이 있을 때 원인 후보 검색

Perplexity는 프로젝트 내부 코드만으로 판단하기 어려운 외부 정보 확인에 사용한다.

## 7.8 Gemini와 Claude의 실무 구분 기준

새 작업단계 설계안 검토, MVP 단계 분할 검토, UI·저장·전투·Android 회귀 리스크 검토, 테스트 실패 원인 후보 정리에는 Gemini를 우선 제안한다.

작업지시문 명확성 검토, 보완지시문 충돌 여부 검토, 폐기 또는 보완 판단의 논리 검토, 최종 구현지시문 레드팀 검토에는 Claude를 우선 제안한다.

실제 코드 구조 확인이 필요한 외부 코드리뷰에는 ClaudeCode를 우선 제안한다.

최신 Unity, Android, Gradle, SDK, 패키지 정보 검색에는 Perplexity를 우선 제안한다.

# 8. 공통 예외 처리 규격

## 8.1 본 장의 성격

공통 예외 처리 규격은 작업단계 표준 흐름을 대체하지 않는다.

각 단계에서 자주 발생하는 예외는 `4. 작업단계 표준 흐름`의 해당 단계에 직접 따른다.

본 장은 현재 단계가 불명확하거나 여러 단계에 걸친 예외 상황에서 GPT가 참고하는 보조 판단 기준이다.

## 8.2 현재 단계 우선 확인

예외 상황이 발생하면 GPT는 먼저 현재 작업단계, 현재 브랜치, 마지막 정상 완료 단계, 마지막 정상 커밋 또는 문서 기준을 확인한다.

현재 단계가 불명확하면 다음 작업으로 넘어가지 않고 상태 확인을 먼저 진행한다.

## 8.3 사용자 제공 결과 우선

사용자가 제공한 터미널 결과, Unity 테스트 결과, Android 테스트 결과, 완료보고, AI 검토 결과는 GPT 판단의 우선 근거로 삼는다.

완료보고가 정상이라고 되어 있어도 Unity 또는 Android 테스트에서 실패하면 테스트 결과를 우선한다.

## 8.4 커밋 전 안전 원칙

Git 상태가 불명확하거나 테스트 결과가 통과되지 않았거나 커밋 대상이 확인되지 않은 경우, GPT는 커밋, push, merge, branch delete 명령어를 제시하지 않는다.

## 8.5 문서 확인 실패 시 원칙

Drive 문서를 찾지 못하면 파일명, 저장 위치, 문서 생성 여부, 문서번호, Drive 검색 반영 지연 가능성을 먼저 확인한다.

마지막 수단으로만 사용자가 문서 원문을 채팅에 직접 붙여넣도록 요청한다.

## 8.6 원인 판단 불명확 시 원칙

GPT 단독으로 문제 원인을 판단하기 어려우면 Gemini, Claude, Perplexity 협업을 제안한다.

기술적 원인 후보와 회귀 리스크 검토는 Gemini를 우선 제안한다.

판단 논리, 작업지시문 충돌, 보완 또는 폐기 판단 검토는 Claude를 우선 제안한다.

외부 정보 확인이 필요한 최신 Unity, Android, Gradle, SDK, 패키지, OS 관련 문제는 Perplexity를 우선 제안한다.

## 8.7 보완과 폐기 판단 원칙

기존 작업 목표와 브랜치를 유지한 채 수정 가능한 문제는 보완루프로 처리한다.

작업 결과를 신뢰하기 어렵거나 이전 정상 커밋 상태로 되돌리는 것이 안전하면 폐기결정을 제안한다.

폐기된 작업단계번호는 재사용하지 않는다.

## 8.8 강제 명령 금지

원인과 현재 상태가 명확하지 않은 상황에서 GPT는 강제 push, 강제 reset, 강제 checkout, 강제 삭제 명령을 임의로 제시하지 않는다.

필요 시 먼저 상태 확인 명령을 제시하고, 사용자의 결과를 확인한 뒤 다음 명령을 안내한다.

## 8.9 사용자 결정 필요 상황

보완루프 진입, 폐기결정, 추가 AI검토, Perplexity 검색, 다음 단계 진입처럼 사용자 결정이 필요한 상황에서는 GPT가 임의로 진행하지 않고 선택지를 제시한 뒤 사용자의 결정을 따른다.

# 9. 새 채팅에서 이어갈 때의 기준

## 9.1 본 장의 성격

새 채팅에서 이어갈 때의 기준은 표준흐름에 바로 진입하기 전의 상태 복구 절차다.

GPT는 새 채팅에서 현재 작업단계와 표준흐름 위치를 확인한 뒤, 사용자에게 재확인을 받고 표준흐름으로 복귀한다.

## 9.2 목적

새 채팅에서 Anima Spire 작업을 이어갈 때 GPT는 바로 구현지시, 검토요청서, 검색요청서, Git 명령어를 작성하지 않는다.

새 채팅의 첫 목표는 현재 작업 상태를 복구하고, 표준흐름의 어느 단계로 복귀해야 하는지 판단하는 것이다.

특히 이전 채팅이 오류, 기능 고장, 과도한 길이, 연결 문제 등으로 인해 정상적인 인수인계 없이 종료된 경우, GPT는 Drive 문서와 GitHub 코드 상태를 기준으로 작업 흐름을 복구해야 한다.

## 9.3 새 채팅 진입 시 우선 확인 항목

GPT는 새 채팅에서 작업을 이어갈 때 다음 항목을 우선 확인한다.

1. 현재 MVP 번호와 MVP명
2. 현재 작업단계번호와 작업명
3. 현재 작업단계의 진행 위치
4. 현재 Git 브랜치
5. 최신 main 커밋 또는 직전 정상 완료 커밋
6. 직전 완료 작업단계
7. 현재 작업단계에서 이미 생성된 Drive 문서
8. 현재 작업단계의 작업차수
9. 현재 작업이 최초 흐름인지, 보완루프인지, 폐기 후 재진입인지
10. 사용자가 원하는 다음 행동

## 9.4 사용자가 충분한 인수인계 정보를 제공한 경우

사용자가 새 채팅 첫 메시지에서 현재 MVP, 작업단계, 브랜치, 직전 완료 단계, 현재까지 생성된 문서명, Git 상태, 다음에 해야 할 작업을 충분히 제공한 경우, GPT는 그 내용을 우선 기준으로 삼는다.

이 경우 GPT는 불필요하게 처음부터 다시 조사하지 않고, 사용자가 제공한 정보를 바탕으로 짧게 상태를 요약한 뒤 다음 단계로 진입할 수 있다.

단, 제공된 정보 사이에 충돌이 있거나 현재 상태가 불명확한 경우에는 바로 다음 단계로 넘어가지 않고 충돌 지점을 먼저 확인한다.

## 9.5 사용자가 충분한 정보를 제공하지 않은 경우

사용자가 “이어서 진행하자”, “아까 작업 이어가자”, “작업 재개하자” 정도만 말하고 충분한 인수인계 정보를 제공하지 않은 경우, GPT는 Drive 문서와 GitHub 코드 상태를 기준으로 작업 흐름을 복구한다.

이 경우 GPT는 다음 순서로 확인한다.

1. 프로젝트 소스 또는 GitHub의 `GPT_WORK_RULES.md` 확인
2. Drive의 `01_전체기획` 문서 확인
3. Drive의 최신 문서번호 문서 확인
4. 최신 문서 기준 역순 15개 내외 문서 확인
5. GitHub 최신 상태 및 관련 코드 확인
6. 작업 재개 가능 상태 브리핑

GPT는 프로젝트 소스 또는 GitHub의 `GPT_WORK_RULES.md`를 먼저 확인하여 현재 적용해야 할 GPT 운영 기준을 파악한다.

그 다음 Drive의 `01_전체기획` 문서를 확인하여 Anima Spire 프로젝트의 전체 기획 방향, 현재 게임 구조, 주요 금지사항, 현재 명칭 기준을 파악한다.

이후 Drive 내 문서번호 기준으로 가장 마지막 일련번호 문서를 확인한다.

해당 문서가 `02_카테고리별_기획`, `03_MVP로드맵`, `04_AI협업문서`, `05_인수인계`, `07_버그_이슈_테스트기록` 등 어느 흐름에 속하는지 파악하고, 최신 작업 맥락을 확인한다.

가장 마지막 일련번호 문서를 기준으로 역순으로 최근 약 15개 문서를 확인하여, 최근 작업 흐름, 현재 MVP, 직전 완료 단계, 현재 진행 중이던 단계, 보완루프 또는 폐기 여부를 파악한다.

Drive 문서 기준으로 현재 작업 흐름을 파악한 뒤, GitHub의 최신 코드 상태와 관련 파일을 확인하여 문서 내용과 코드 구조가 일치하는지 검토한다.

GPT는 확인한 내용을 바탕으로 현재까지 숙지한 사항을 사용자에게 짧게 브리핑하고, 작업 재개 준비가 되었음을 안내한다.

## 9.6 Drive 문서 확인 기준

Drive 문서 확인이 필요한 경우, GPT는 다음 순서로 확인한다.

1. `01_전체기획` 폴더의 핵심 기획 문서
2. 문서번호 기준 가장 마지막 일련번호 문서
3. 최신 문서가 속한 `02_카테고리별_기획`, `03_MVP로드맵`, `04_AI협업문서`, `05_인수인계`, `07_버그_이슈_테스트기록` 흐름
4. 최신 문서 기준 역순 15개 내외 문서
5. 현재 작업단계의 구현지시, 검토세트, 검색세트, 폐기결정 문서
6. 필요 시 해당 MVP 폴더의 최근 문서

문서번호가 있는 경우에는 문서번호가 큰 문서를 최신 문서 후보로 본다.

다만 문서번호와 생성일만으로 내용을 단정하지 않고, 실제 문서 내용을 확인하여 현재 흐름을 판단한다.

## 9.7 GitHub 코드 확인 기준

Drive 문서 확인을 통해 현재 작업 흐름을 파악한 뒤, GPT는 GitHub 최신 상태와 관련 코드 파일을 확인한다.

GitHub 코드 확인의 목적은 다음과 같다.

1. Drive 문서상 완료된 구현이 실제 코드에 반영되어 있는지 확인
2. 현재 작업단계와 관련된 파일 구조 확인
3. GPT가 설계안 또는 작업지시문을 작성하기 전에 실제 코드 구조를 파악
4. Drive 문서와 GitHub 코드 상태가 서로 충돌하지 않는지 확인

다만 사용자가 이미 충분한 로컬 diff, Git 로그, Codex 완료보고, Unity 테스트 결과를 제공한 경우에는 GitHub 코드 확인을 생략할 수 있다.

이 경우 GPT는 코드 확인을 생략했음을 명시하고, 사용자가 제공한 자료 기준으로 판단한다.

## 9.8 새 채팅에서 표준흐름 복귀 기준

상태 확인이 끝나면 GPT는 현재 작업이 표준흐름 중 어느 단계에 해당하는지 예상하여 사용자에게 브리핑한다.

다만 GPT는 이 판단을 최종 확정하지 않고, 사용자에게 재확인을 요청한다.

브리핑 형식은 다음 기준을 따른다.

`GPT 작업기준, Drive 내 기획서 및 최근 문서, GitHub 코드 구조를 확인했습니다. 현재 작업흐름은 {작업단계번호}_{작업명}의 {현재 표준흐름 단계}로 예상됩니다. 다음 단계는 {다음 단계명}으로 보입니다. 이 판단이 맞는지 확인해 주시기 바랍니다.`

예시는 다음과 같다.

`GPT 작업기준, Drive 내 기획서 및 최근 문서, GitHub 코드 구조를 확인했습니다. 현재 작업흐름은 033F_Android상태바회귀보완 작업단계의 v02 보완루프 이후 테스트 대기 상태로 예상됩니다. 다음 단계는 [테스트]로 보입니다. 이 판단이 맞는지 확인해 주시기 바랍니다.`

사용자가 확인하면 GPT는 해당 표준흐름 단계로 복귀한다.

사용자가 아니라고 하면, GPT는 불일치 지점을 다시 확인하고 현재 상태를 재정리한다.

## 9.9 상태 불일치 시 처리 기준

새 채팅에서 확인한 정보가 서로 맞지 않는 경우, GPT는 바로 다음 단계로 넘어가지 않는다.

상태 불일치 예시는 다음과 같다.

1. 사용자가 말한 현재 단계와 Drive 최신 문서가 다름
2. 사용자가 말한 브랜치와 Git 로그가 다름
3. 완료보고는 있으나 테스트 결과가 없음
4. 테스트 실패가 있었으나 보완루프 또는 폐기결정 기록이 없음
5. 폐기결정이 있었으나 같은 단계번호를 다시 사용하려 함
6. Git 정리 완료라고 했으나 main 또는 clean 상태 확인이 없음

이 경우 GPT는 불일치 지점을 먼저 요약하고, 현재 상태를 확정하기 위한 확인 항목 또는 명령어를 제시한다.

## 9.10 새 채팅에서 금지되는 진행

새 채팅에서 현재 상태가 명확하지 않은 경우 GPT는 다음 작업을 하지 않는다.

1. 바로 구현지시문 작성
2. 바로 검토요청서 작성
3. 바로 검색요청서 작성
4. 바로 커밋 명령어 작성
5. 바로 브랜치 삭제 명령어 작성
6. 현재 작업단계 종료 선언
7. 다음 작업단계 진입 선언
8. 폐기 또는 보완 확정 선언

상태가 명확하지 않을 때 GPT는 먼저 상태 확인, 문서 확인, GitHub 코드 확인, Git 상태 확인을 진행한다.

## 9.11 새 채팅 시작 브리핑 형식

상태 확인이 끝난 경우 GPT는 짧은 브리핑을 제공한다.

브리핑에는 다음 내용을 포함한다.

1. 확인한 GPT 작업기준
2. 확인한 핵심 기획 문서
3. 확인한 최근 문서 범위
4. 현재 MVP
5. 현재 작업단계
6. 현재 브랜치 또는 Git 기준 상태
7. 직전 완료 단계
8. 현재 표준흐름 위치
9. 다음에 진행할 단계
10. 사용자 확인 요청

예시는 다음과 같다.

`GPT 작업기준과 Drive의 01_전체기획 문서, 최신 일련번호 기준 최근 15개 문서, GitHub 관련 코드 구조를 확인했습니다. 현재 상태는 033F_Android상태바회귀보완 작업단계의 v02 보완루프 이후 테스트 대기 상태로 판단됩니다. 현재 브랜치는 feature/033f-android-ui-fix 기준으로 보이며, 직전 정상 완료 단계는 033E입니다. 다음 흐름은 [테스트] 단계로 보입니다. 이 판단이 맞는지 확인해 주시기 바랍니다.`

## 9.12 본 장의 핵심 원칙

새 채팅에서 GPT는 작업을 새로 시작하는 것이 아니라, 현재 작업 상태를 복구한 뒤 표준흐름의 정확한 위치로 복귀해야 한다.

이전 채팅의 인수인계가 없더라도, GPT는 프로젝트 소스 또는 GitHub의 `GPT_WORK_RULES.md`, Drive의 전체기획 문서, 최신 문서번호 기준 최근 문서, GitHub 코드 구조를 확인하여 현재 작업 흐름을 복구한다.

현재 단계, 브랜치, 문서, 테스트 상태, Git 상태가 명확하지 않으면 GPT는 다음 단계로 진행하지 않는다.

# 10. 프로젝트 문서 작성 규격

## 10.1 본 장의 성격

본 장은 AI협업 문서가 아닌 Anima Spire 프로젝트 문서를 작성하거나 저장 위치를 안내할 때 GPT가 따라야 할 기준을 정의한다.

본 장의 적용 대상은 기획문서, 카테고리별 기획문서, MVP 구현로드맵, MVP 구현로드맵 완료반영판, 인수인계 문서, 폐기결정 문서, 빌드·릴리즈 관련 문서, 아카이브 처리 대상 문서다.

검토세트, 검색세트, 구현지시, 검토지시는 본 장이 아니라 `5. 파일명 및 문서 내부규격`과 `6. 문서 유형별 템플릿`을 따른다.

본 장은 프로젝트 문서 체계를 설명하기 위한 장이 아니라, GPT가 프로젝트 문서를 새로 작성하거나 개정할 때 저장 위치, 파일명, 문서 넘버링, 우선순위를 혼동하지 않도록 하기 위한 최소 가드레일이다.

## 10.2 적용 시점과 소급적용 금지

본 장의 프로젝트 문서 작성 규격은 GPT작업기준 v04 확정 이후 새로 작성하거나 개정하는 문서부터 적용한다.

기존 과거 문서의 파일명, 번호체계, 저장 위치에는 소급 적용하지 않는다.

과거 `v0.x` 체계로 작성된 문서는 그대로 유지한다.

과거 폴더 구조에서 생성된 문서도 현행 폴더 체계에 맞추어 전부 재작성하거나 이름을 바꾸지 않는다.

다만 현행 체계와 충돌하는 구버전 문서, 폐지 폴더 문서, 정식 문서에 통합 완료된 임시문서는 삭제하지 않고 `99_아카이브`로 이동한다.

## 10.3 Drive 폴더 역할 기준

Drive는 기획문서, 로드맵, AI협업문서, 인수인계, 테스트·이슈 기록, 빌드·릴리즈 산출물, 아카이브 문서를 보관하는 작업 자료 공간이다.

Drive의 현행 주요 폴더 역할은 다음 기준을 따른다.

`01_전체기획`은 게임의 상위 기획, First Release 범위, 최종 서비스비전을 관리한다.

`02_카테고리별_기획`은 전투, 장비, UI, 마법스크롤, 연구실 등 시스템별 상세 기획문서를 관리한다.

`03_MVP로드맵`은 MVP 구현로드맵과 MVP 구현로드맵 완료반영판을 관리한다.

`04_AI협업문서`는 검토세트, 검색세트, 구현지시, 검토지시를 관리한다.

`05_인수인계`는 새 채팅 또는 작업 이관을 위한 인수인계 문서를 관리한다.

`07_버그_이슈_테스트기록`은 버그, 테스트 기록, 폐기결정 문서를 관리한다.

`08_빌드_릴리즈`는 APK 등 빌드 산출물을 관리한다.

`99_아카이브`는 구버전 문서, 폐지 폴더 문서, 정식 문서에 통합 완료된 문서, 현행 기준에서 활성 참조가 끝난 문서를 관리한다.

## 10.4 프로젝트 문서 파일명 규격

프로젝트 문서는 자연어형 파일명을 사용한다.

AI협업문서처럼 모든 구간을 언더바로 분리하는 엄격한 기계식 파일명을 적용하지 않는다.

프로젝트 문서의 기본 파일명 형식은 다음 기준을 따른다.

YYMMDD_NN_Anima Spire {문서명}{넘버링}

`YYMMDD_NN`은 문서 생성 시점의 한국 기준 날짜와 해당 날짜 내 문서 생성 순서로 부여한다.

`문서명`은 사용자가 검색하고 읽기 쉬운 자연어형 문서명으로 작성한다.

기획문서의 r 계열 넘버링은 문서명 뒤에 언더바로 붙인다.

예시는 다음과 같다.

260601_03_Anima Spire 전체 기획서_r033.01

260607_01_Anima Spire UI 기획서_r033.01

v 계열 문서는 문서명 뒤에 공백으로 버전을 붙인다.

예시는 다음과 같다.

260610_01_Anima Spire 033 마법스크롤 기초 구조 MVP 구현로드맵 v01

260620_02_Anima Spire 033 마법스크롤 기초 구조 MVP 구현로드맵 완료반영판 v01

인수인계 문서는 반복 생성과 검색 편의를 위해 다음 형식을 사용한다.

YYMMDD_NN_Anima Spire 인수인계_{인수인계주제}_vXX

예시는 다음과 같다.

260615_04_Anima Spire 인수인계_033마법스크롤MVP진행중_v01

프로젝트 문서를 GPT가 새로 작성하거나 개정 본문을 출력하는 경우, 출력 형식은 다음 기준을 따른다.

먼저 문서 제목 또는 파일명을 코드블록으로 작성한다.

그 다음 문서 본문은 마크다운 렌더링 문서 블록으로 작성한다.

문서 제목 또는 파일명 코드블록은 사용자가 Drive 또는 로컬 파일명으로 바로 복사할 수 있게 하기 위한 것이다.

문서 본문은 사용자가 그대로 복사하여 Drive 문서 또는 Markdown 문서에 붙여넣을 수 있도록 마크다운 렌더링 문서 블록으로 작성한다.

이 기준은 기획문서, 카테고리별 기획문서, MVP 구현로드맵, MVP 구현로드맵 완료반영판, 인수인계 문서 등 프로젝트 문서 작성 시 적용한다.

검토세트, 검색세트, 구현지시, 검토지시, 폐기결정 문서는 본 기준이 아니라 `5. 파일명 및 문서 내부규격`과 `6. 문서 유형별 템플릿`의 출력 기준을 따른다.

## 10.5 r 계열 / v 계열 넘버링 기준

프로젝트 문서의 넘버링은 r 계열과 v 계열로 구분한다.

r 계열은 revision의 의미로 사용한다.

r 계열은 `01_전체기획`과 `02_카테고리별_기획` 폴더의 기획문서에 적용한다.

r 계열의 형식은 다음과 같다.

r{MVP번호}.{개정순번}

예시는 다음과 같다.

r033.01

r033.02

r034.01

`r033.01`은 033 MVP 기준 시점에서 해당 기획문서의 첫 번째 개정을 의미한다.

같은 MVP 진행 중 같은 기획문서를 다시 개정하면 `r033.02`, `r033.03`처럼 개정순번을 증가시킨다.

다음 MVP 시점에서 개정하면 MVP 번호를 바꾸고 개정순번은 다시 01부터 시작한다.

MVP 진행 중 기획문서를 개정하는 경우 현재 진행 중인 MVP 번호를 사용한다.

MVP 완료 후 다음 MVP 시작 전 기획문서를 개정하는 경우 마지막으로 완료된 MVP 번호를 사용한다.

v 계열은 version의 의미로 사용한다.

v 계열은 기획문서를 제외한 나머지 문서에 적용한다.

MVP 구현로드맵, MVP 구현로드맵 완료반영판, 인수인계, 폐기결정, AI협업문서는 v 계열을 사용한다.

단, AI협업문서의 vXX는 동일 작업단계 내 작업차수를 의미한다.

그 외 프로젝트 운영문서의 vXX는 문서 버전 또는 문서 차수를 의미한다.

## 10.6 MVP 구현로드맵 운영 기준

MVP 구현로드맵은 카테고리별 기획 내용을 실제 구현 가능한 MVP 단위와 작업단계로 쪼개기 위한 실행 문서다.

MVP 구현로드맵의 정식 명칭은 `MVP 구현로드맵`으로 한다.

MVP 완료 문서의 정식 명칭은 `MVP 구현로드맵 완료반영판`으로 한다.

MVP 구현로드맵 파일명은 다음 형식을 따른다.

YYMMDD_NN_Anima Spire {MVP번호} {MVP명} MVP 구현로드맵 vXX

MVP 구현로드맵 완료반영판 파일명은 다음 형식을 따른다.

YYMMDD_NN_Anima Spire {MVP번호} {MVP명} MVP 구현로드맵 완료반영판 vXX

MVP 구현로드맵은 최초 작성 시 v01로 작성한다.

MVP 진행 중 중대한 변수로 기존 로드맵이 더 이상 작업 기준으로 적합하지 않으면 v02, v03으로 버전업한다.

사소한 작업단계 세분화는 로드맵 버전업 사유로 보지 않는다.

예를 들어 033C로 계획했으나 구현 중 033C-1, 033C-2로 세분화하는 정도는 로드맵 버전업 없이 진행할 수 있다.

로드맵 버전업 시 `YYMMDD_NN`은 새 문서 생성 시점의 한국 기준 날짜와 해당 날짜 내 문서 생성 순서에 따라 새로 부여한다.

로드맵 버전업 시 문서명 본체는 유지하고, 마지막 버전 표기만 v01에서 v02, v03으로 증가시킨다.

새 버전의 MVP 구현로드맵이 작성되면 기존 버전은 현행 `99_아카이브` 체계로 이동한다.

MVP 구현로드맵 완료반영판은 해당 MVP 완료 후 작성한다.

완료반영판이 작성되면 해당 MVP의 최종 상태 기준은 완료반영판이다.

기존 MVP 구현로드맵은 계획 기록으로 유지하되, 완료반영판과 충돌하면 완료반영판을 우선한다.

진행 중이거나 완료된 MVP는 `03_MVP로드맵` 안에서 MVP 번호별 하위폴더로 관리한다.

아직 개발 단계에 진입하지 않았지만 빠른 시일 내 개발 가능성이 있어 먼저 작성한 예정 MVP 구현로드맵은 `03_MVP로드맵` 루트폴더에 임시보관한다.

해당 MVP가 실제 개발 대상으로 확정되면 MVP 번호별 하위폴더를 생성하고, 루트폴더의 예정 MVP 구현로드맵을 해당 하위폴더로 이동한다.

## 10.7 인수인계 문서 기준

인수인계 문서는 새 채팅에서 작업 상태를 정확히 복구하기 위한 상태 스냅샷 문서다.

인수인계 문서는 `05_인수인계` 폴더에 저장한다.

인수인계 문서 파일명은 다음 형식을 따른다.

YYMMDD_NN_Anima Spire 인수인계_{인수인계주제}_vXX

인수인계 문서는 다음 상황에서 작성한다.

1. MVP가 완료된 경우
2. MVP 도중 채팅이 길어져 새 채팅으로 이관해야 하는 경우
3. 작업 흐름이 복잡해져 상태 정리가 필요한 경우
4. 보완루프, 테스트 이슈, 로드맵 재정의 등으로 다음 GPT가 맥락을 잃을 우려가 있는 경우
5. 사용자가 요청한 경우

인수인계 문서에는 최소한 다음 항목을 포함한다.

1. 현재 MVP
2. 현재 작업단계
3. 현재 표준흐름 위치
4. 현재 브랜치와 Git 상태
5. 최근 완료 내용
6. 남은 작업
7. 우선 확인할 문서
8. 테스트 상태
9. 주의사항
10. 다음 채팅 시작 프롬프트

인수인계 문서는 단순한 첫 채팅 프롬프트로 대체하지 않는다.

다만 인수인계 문서 안에 다음 채팅 시작 프롬프트를 포함하여, 사용자가 새 채팅 첫 메시지로 바로 사용할 수 있게 한다.

## 10.8 폐기결정 및 99_아카이브 기준

폐기결정 문서는 테스트 실패, 보완루프 실패, 구조적 오류, 구현 결과 불신뢰 등으로 특정 작업단계의 결과를 폐기하기로 결정한 기록이다.

폐기결정 문서는 `07_버그_이슈_테스트기록 > 07-01_폐기결정`에 저장한다.

폐기결정 문서의 파일명은 `5. 파일명 및 문서 내부규격`의 폐기결정 문서 파일명 규격을 따른다.

구버전 문서, 폐지 폴더 문서, 정식 문서에 통합 완료된 문서, 현행 기준에서 활성 참조가 끝난 문서는 삭제하지 않고 `99_아카이브`로 이동한다.

과거 AI 검토 원문과 검토통합정리 문서는 참고 기록이며, 최신 기획문서, MVP 구현로드맵, MVP 구현로드맵 완료반영판, GitHub main 구현 상태보다 우선하지 않는다.

아카이브 구조는 현행 체계를 유지한다.

추후 실제 운영 중 아카이브 위치나 구조로 인한 불편이 반복될 경우, 그 시점에 구조 변경을 재검토한다.
