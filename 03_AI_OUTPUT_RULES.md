# AI Output Rules - Anima Spire 출력기준_v02

## 1. 공통 출력 원칙

### 1.1 문서 목적

이 문서는 Anima Spire 프로젝트에서 GPT, Claude, Gemini, Perplexity 등 협업 AI가 산출물을 작성할 때 적용하는 출력기준 문서다.

이 문서는 판단기준 문서가 아니다.

현재 요청의 작업 유형, 담당 AI, 사용자 결정 필요 여부, 보완·폐기·방향점검 전환 여부는 `02_AI_WORKFLOW_RULES.md`를 따른다.

이 문서는 판단이 끝난 뒤 다음을 정한다.

1. 어떤 형식으로 응답할지
2. 어떤 파일명을 사용할지
3. 어떤 문서블록·코드블록·채팅 직접출력 기준을 적용할지
4. 어떤 문서 템플릿을 사용할지
5. Drive 저장 위치를 어떻게 안내할지
6. AI협업문서와 개발문서를 어떤 구조로 출력할지

구현자 또는 코드검토자 가드레일은 `IMPLEMENTATION_GUARDRAILS.md`를 따른다.

### 1.2 적용 대상

이 문서는 다음 경우에 적용한다.

1. 파일명을 생성하는 Drive 저장문서를 작성하는 경우
2. 사용자가 저장하거나 다른 AI에게 전달할 문서 본문을 작성하는 경우
3. Claude, Gemini, Perplexity, Codex, ClaudeCode에 보낼 발신문서를 작성하는 경우
4. AI 회신 형식을 발신문서 안에 명시해야 하는 경우
5. 코드블록, 문서블록, 채팅 직접출력 방식을 구분해야 하는 경우
6. Drive 저장 위치를 안내해야 하는 경우
7. 정규문서, 인수인계, 폐기결정, MVP 구현로드맵, 완료반영판을 작성하는 경우
8. 터미널 명령어 또는 Git 후속 안내를 출력하는 경우

### 1.3 비적용 대상

파일명을 생성하지 않는 일반 응답, 간단한 판단, 짧은 문구 수정, 단순 질의응답은 이 문서의 문서 출력규격을 강제 적용하지 않는다.

단, 사용자가 특정 출력형식, 문서블록, 코드블록, 파일명, 저장 위치를 명시적으로 요청한 경우에는 이 문서의 해당 장을 따른다.

### 1.4 언어와 말투

Anima Spire 프로젝트 응답은 원칙적으로 한국어 존대말로 작성한다.

다만 `IMPLEMENTATION_GUARDRAILS.md`, Codex 구현지시문, ClaudeCode 검토지시문 등 구현자용 영문 가드레일 또는 영문 지시가 필요한 문서는 영어로 작성할 수 있다.

응답은 실무 작업 흐름에 맞게 명확하고 차분하게 작성한다.

줄임말, 이모티콘, 과한 감탄 표현은 사용하지 않는다.

### 1.5 확인 불가 항목 출력 기준

확인하지 않은 항목은 확정 표현으로 쓰지 않는다.

확인할 수 없는 항목은 다음 중 하나로 분리한다.

1. 확인 완료
2. 확인 필요
3. 사용자 제공 필요
4. 보류
5. 추정 금지

AI는 확인 불가 항목을 문서 본문에 확정 사실처럼 포함하지 않는다.

### 1.6 제안과 결정의 구분

문서나 응답에서 다음 상태를 구분하여 표현한다.

1. 결정
2. 제안
3. 보류
4. 제외
5. 확인 필요

사용자가 명시적으로 확정하지 않은 항목은 “결정”으로 쓰지 않는다.

## 2. 일반 응답 출력 형식

### 2.1 판단·제안 응답 기본 형식

AI가 판단, 방향 제안, 작업 흐름 전환, 사용자 결정 요청을 포함하는 응답을 할 때에는 원칙적으로 다음 형식을 따른다.

```text
[상황 판단]
현재 상황이 어떤 기준에 해당하는지 간단히 정리한다.

[제안 내용]
제안이 하나면 단일안으로 제시한다.
제안이 둘 이상이면 A안 / B안 / C안으로 병렬 제시한다.
각 안의 핵심 장단점을 짧게 정리한다.

[결정 필요 항목]
□ 항목 1
□ 항목 2
□ 항목 3

[비고]
추가 확인 필요사항, 판단하지 못한 부분, 충돌 가능성만 짧게 적는다.
없으면 생략한다.
```

### 2.2 기본 형식을 생략할 수 있는 경우

다음 경우에는 2.1 형식을 생략할 수 있다.

1. 사용자가 단일 사실만 묻는 경우
2. 사용자가 예/아니오 답변만 요청한 경우
3. 사용자가 명시적으로 본문만 요청한 경우
4. 간단한 문구 수정 요청인 경우
5. 이미 확정된 내용을 단순 정리하는 경우
6. 파일명을 생성하지 않는 자유 응답인 경우

### 2.3 기본 형식을 반드시 적용하는 경우

다음 경우에는 2.1 형식을 적용한다.

1. 판단이 포함된 응답
2. 방향 제안이 포함된 응답
3. 작업 흐름 전환이 포함된 응답
4. Claude 방향점검 트리거 발동 시
5. 폐기, 보완, 로드맵 재설정 중 선택을 제안하는 경우
6. 기준문서 개정 방향을 제안하는 경우
7. 사용자 결정이 필요한 항목이 있는 경우

### 2.4 다음 단계 안내 형식

모든 응답 말미에 다음 단계 블록을 기계적으로 반복하지 않는다.

다음 단계 안내는 다음 기준을 따른다.

1. 새 다음 단계가 생겼을 때 1회 안내한다.
2. 사용자가 즉시 진행하지 않고 다른 대화가 이어지면 반복하지 않는다.
3. 보류된 다음 단계가 2개 이상이면 모두 제시하고 우선순위 결정을 요청한다.
4. 현재 주제가 마무리되었다고 판단되면 추가 논의 또는 다음 단계 안내 여부를 묻는다.

다음 단계 안내 형식은 다음을 따른다.

```text
**다음 단계**
담당: GPT / Claude / Gemini / Perplexity / Codex / ClaudeCode / 사용자
할 일: 한 줄로 작성
필요한 것: 필요한 경우에만 작성
```

다음 단계 안내는 사용자의 명시적 결정을 대체하지 않는다.

## 3. 문서블록·코드블록·채팅 직접출력 기준

### 3.1 적용 기준

파일명을 생성하는 Drive 저장문서 또는 AI 간 발신·회신 세트문서에는 이 장의 출력규격을 적용한다.

파일명을 생성하지 않는 일반 응답, 간단한 판단, 짧은 문구 수정, 단순 질의응답은 이 장의 출력규격을 강제하지 않는다.

### 3.2 파일명 출력 기준

파일명을 생성하는 모든 문서의 파일명은 코드블록으로 작성한다.

파일명은 일반 문장 안에만 흘려 쓰지 않고, 사용자가 그대로 복사할 수 있도록 별도 코드블록으로 제공한다.

예시는 다음과 같다.

```text
260618_01_Anima Spire 운영_Claude_기안문_기준문서4종체계개편_v01
```

여러 파일명을 제시하는 경우에도 코드블록 안에 한 줄씩 작성한다.

```text
01_AI_ROUTER.md
02_AI_WORKFLOW_RULES.md
03_AI_OUTPUT_RULES.md
IMPLEMENTATION_GUARDRAILS.md
```

### 3.3 문서 본문 출력 기준

파일명을 생성하는 단독문서와 발신문서의 본문은 마크다운 렌더링이 적용되는 문서블록으로 작성한다.

문서 본문 전체를 `markdown` 코드블록 안에 넣지 않는다.

문서 본문은 사용자가 바로 읽고 검토할 수 있도록 마크다운 헤딩, 표, 목록이 렌더링되는 형태로 작성한다.

문서 본문에 포함되는 터미널 명령어, 파일명 예시, 코드 예시는 필요한 경우 문서 본문 내부에서 코드블록으로 작성할 수 있다.

### 3.4 단독문서 출력 기준

다음 문서는 파일명 코드블록과 문서블록 본문으로 작성한다.

1. 정규문서
2. 인수인계 문서
3. 폐기결정 문서
4. MVP 구현로드맵
5. MVP 구현로드맵 완료반영판
6. 기준문서 본문
7. 테스트·이슈기록
8. 빌드·릴리즈 기록

### 3.5 발신문서 출력 기준

다음 문서는 파일명 코드블록과 문서블록 본문으로 작성한다.

1. Claude 기안 요청문
2. Claude 방향점검 요청문
3. Gemini 검토요청문
4. Perplexity 검색요청문
5. Codex 구현지시문
6. ClaudeCode 검토지시문
7. ClaudeCode 구현지시문

발신문서에는 회신 출력규격을 반드시 포함한다.

Gemini, Perplexity, ClaudeCode, Codex 등 외부 AI 또는 구현자가 회신 형식을 지키지 않을 가능성이 있으므로, GPT가 작성하는 발신문서에는 답변 제목, 헤딩 규격, 부연설명 금지, 확인 불가 처리 기준을 명시한다.

### 3.6 회신문서 출력 기준

AI 간 세트문서의 회신은 부연설명 없이 본문 내용만 채팅에 직접 출력하도록 요청한다.

다음은 회신문서에 해당한다.

1. Claude 기안 답변
2. Claude 방향점검 답변
3. Gemini 검토답변
4. Perplexity 검색결과
5. Codex 완료보고
6. ClaudeCode 검토답변
7. ClaudeCode 구현 완료보고

회신문서는 별도의 파일명을 생성하지 않는다.

사용자는 해당 회신 전문을 Drive 문서에 구분선을 삽입한 뒤 직접 붙여넣는다.

나중에 필요한 경우 답신 전문을 다시 확인할 수 있도록, 회신 전문은 요약본이 아니라 원문 전체를 보존하는 것을 원칙으로 한다.

### 3.7 AI 간 세트문서의 송신·회신 구분

AI 간 주고받는 세트문서는 송신 문서와 회신 문서를 구분한다.

송신 문서는 사용자가 다른 AI에게 전달하기 위한 요청문이다.

송신 문서는 파일명 코드블록과 문서블록 본문으로 작성한다.

회신 문서는 다른 AI가 요청에 답하는 결과물이다.

회신 문서는 부연설명 없이 채팅에 직접 출력하도록 요청한다.

사용자는 회신 전문을 Drive 문서 내 구분선 아래에 붙여넣는다.

GPT가 이후 해당 회신을 해석할 때에는 다음을 분리한다.

1. 회신 원문
2. GPT 해석
3. 반영 여부
4. 사용자 결정 필요 항목
5. 보류 또는 제외 항목

### 3.8 헤딩 규격

문서 본문과 AI 회신 본문은 다음 헤딩 규격을 따른다.

첫 줄의 문서 제목 또는 답변 제목은 `#`으로 작성한다.

본문의 1단계 장은 `## 1.` 형식으로 작성한다.

본문의 2단계 장은 `### 1.1` 형식으로 작성한다.

본문의 3단계 장은 필요한 경우 `#### 1.1.1` 형식으로 작성한다.

`#`은 문서 제목 또는 답변 제목에만 사용한다.

본문 장 제목에는 `#`을 사용하지 않는다.

### 3.9 헤딩 예시

이 장의 예시는 코드블록으로 작성한다.

단, 실제 문서 본문 출력은 코드블록이 아니라 문서블록으로 작성한다.

문서 본문 헤딩 예시는 다음과 같다.

```markdown
# [작업지시-GPT]

## 1. 작업 목적

### 1.1 세부 목적

## 2. 작업 범위

### 2.1 포함 범위

### 2.2 제외 범위

## 3. 완료보고 요구사항

### 3.1 필수 보고 항목

### 3.2 금지사항 확인
```

AI 회신 본문 헤딩 예시는 다음과 같다.

```markdown
# [검토답변-Gemini]

## 1. 검토 결과 요약

### 1.1 핵심 판단

## 2. 발견된 리스크

### 2.1 구조 리스크

### 2.2 회귀 리스크

## 3. 반영 권장사항

### 3.1 즉시 반영

### 3.2 보류 가능
```

### 3.10 회신규격 요청문 필수 문구

AI에게 발신문서를 작성할 때에는 다음 취지의 회신규격을 포함한다.

```text
답변은 부연설명 없이 본문 내용만 작성해 주세요.

첫 줄은 다음 형식의 제목으로 작성해 주세요.
# [검토답변-Gemini]

본문은 다음 헤딩 규격을 따라 주세요.
## 1.
### 1.1
#### 1.1.1

확인하지 못한 항목은 추정하지 말고 “확인 불가” 또는 “추가 자료 필요”로 분리해 주세요.
```

AI 종류에 따라 제목 예시는 다음처럼 바꾼다.

```text
# [기안답변-Claude]
# [방향점검답변-Claude]
# [검토답변-Gemini]
# [검색결과-Perplexity]
# [완료보고-Codex]
# [검토답변-ClaudeCode]
# [완료보고-ClaudeCode]
```

## 4. 코드블록 및 터미널 명령어 출력 기준

### 4.1 코드블록 기본 원칙

터미널에 붙여넣을 명령어는 반드시 하나의 코드블록으로 제공한다.

코드, 터미널 명령, 파일 본문, 패치 본문은 서로 다른 코드블록으로 분리한다.

사용자가 그대로 복사해 실행해야 하는 내용에는 불필요한 설명문을 코드블록 안에 넣지 않는다.

### 4.2 Git 명령어 출력 기준

Git 후속작업이 필요한 경우 사용자가 별도로 요청하지 않아도 터미널용 명령어 코드블록을 제공할 수 있다.

단, Git 명령어는 사용자가 직접 실행하는 안내용으로만 제공한다.

Codex와 ClaudeCode에게 Git 저장 이력 변경 작업을 지시하지 않는다.

Git merge 안내 시 Vim 등 편집기가 뜨지 않도록 한다.

merge commit이 필요한 경우 `-m` 옵션을 포함한다.

예시는 다음과 같다.

```bash
git checkout main
git pull --ff-only
git merge --no-ff feature/example-branch -m "Merge feature/example-branch"
git push origin main
```

### 4.3 Git 상태 확인 명령어

커밋 전 확인이 필요한 경우 다음 명령어를 우선 안내한다.

```bash
git status
git branch --show-current
git log -1 --oneline
git diff --stat
```

필요한 경우 상세 diff 확인 명령을 추가한다.

```bash
git diff
```

### 4.4 Android 빌드 관련 출력 기준

Codex와 ClaudeCode에게 Android APK 또는 AAB 빌드를 지시하지 않는다.

Android 실기기 테스트가 필요한 경우 사용자가 직접 빌드·설치·실기기 테스트를 수행해야 함을 안내한다.

Android A영역, Safe Area, 상태바, 하단 시스템 제스처, 카메라홀 관련 검증은 Android 실기기 테스트를 최종 기준으로 안내한다.

## 5. 파일명 기준

### 5.1 소스 기준문서 파일명

Anima Spire 프로젝트의 소스 기준문서는 다음 4종으로 운영한다.

```text
01_AI_ROUTER.md
02_AI_WORKFLOW_RULES.md
03_AI_OUTPUT_RULES.md
IMPLEMENTATION_GUARDRAILS.md
```

`01`, `02`, `03`은 AI가 순차 참조하는 문서에만 붙인다.

`IMPLEMENTATION_GUARDRAILS.md`는 구현자 가드레일로서 별도 병렬 트랙이므로 넘버링하지 않는다.

소스 기준문서 파일명에는 버전을 붙이지 않는다.

버전은 문서 내부 첫 줄 제목과 변경 이력에만 기록한다.

예시는 다음과 같다.

```text
파일명: 02_AI_WORKFLOW_RULES.md
문서 내부 제목: # AI Workflow Rules - Anima Spire 작업판단기준_v02
변경 이력: v01, v02
```

### 5.2 Drive 저장문서 기본 파일명 구조

Drive 저장문서의 기본 파일명은 다음 구조를 따른다.

```text
YYMMDD_NN_Anima Spire {대상}_{AI명}_{문서종류}_{문서제목}_vXX
```

각 요소의 의미는 다음과 같다.

| 요소            | 의미                                                   |
| ------------- | ---------------------------------------------------- |
| `YYMMDD`      | 문서 작성일                                               |
| `NN`          | 해당 날짜의 문서 일련번호                                       |
| `Anima Spire` | 프로젝트명                                                |
| `{대상}`        | MVP 번호, 세부단계번호, 운영, 기획 등 문서 대상                       |
| `{AI명}`       | GPT, Claude, Gemini, Codex, ClaudeCode, Perplexity 등 |
| `{문서종류}`      | 기안문, 검토세트, 검색세트, 구현지시, 검토지시, 인수인계 등                  |
| `{문서제목}`      | 문서 내용을 압축한 짧은 제목                                     |
| `vXX`         | 문서 버전 또는 작업차수                                        |

### 5.3 AI명을 생략할 수 있는 문서

다음 문서는 AI명을 생략할 수 있다.

1. 전체 기획서
2. 카테고리별 기획서
3. MVP 구현로드맵
4. MVP 구현로드맵 완료반영판
5. 폐기결정
6. 인수인계
7. 테스트·이슈기록
8. 빌드·릴리즈 기록
9. 사용자가 직접 작성한 결정 기록

예시는 다음과 같다.

```text
260618_02_Anima Spire 034_UI탭구조완성및씬구조예약_MVP구현로드맵_v01
260618_03_Anima Spire 033_UI Surface및Android A영역대응_MVP구현로드맵_완료반영판_v01
260618_04_Anima Spire 인수인계_033완료및034진입전_v01
```

### 5.4 AI명을 포함하는 문서

다음 문서는 원칙적으로 AI명을 포함한다.

1. Claude 기안문
2. Claude 방향점검 요청문
3. Gemini 검토세트
4. Perplexity 검색세트
5. Codex 구현지시
6. ClaudeCode 검토지시
7. ClaudeCode 구현지시
8. GPT 정리문 또는 GPT 해석문

예시는 다음과 같다.

```text
260618_01_Anima Spire 운영_Claude_기안문_기준문서4종체계개편_v01
260618_05_Anima Spire 034A_Gemini_검토세트_UI탭구조설계검토_v01
260618_06_Anima Spire 034A_Codex_구현지시_UI탭구조MVP_v01
260618_07_Anima Spire 034A_ClaudeCode_검토지시_UI탭구조회귀검토_v01
```

### 5.5 일련번호 기준

Drive 저장문서 파일명 맨 앞에는 해당 날짜의 일련번호를 붙인다.

일련번호는 `01`, `02`, `03`처럼 두 자리로 작성한다.

파일명을 생성해야 하는 경우 AI는 먼저 Drive에서 해당 날짜의 기존 문서를 검색한다.

검색 기준은 파일명 맨 앞의 `YYMMDD`이다.

예를 들어 2026년 6월 18일에 작성하는 문서라면 `260618`로 시작하는 기존 문서를 검색한다.

검색 결과에서 가장 큰 일련번호 `NN`을 확인한 뒤, 새 문서에는 다음 번호를 사용한다.

예시는 다음과 같다.

```text id="i7xjbe"
검색 결과:
260618_01_Anima Spire 운영_Claude_기안문_기준문서4종체계개편_v01
260618_02_Anima Spire 034_UI탭구조완성및씬구조예약_MVP구현로드맵_v01
260618_03_Anima Spire 인수인계_033완료및034진입전_v01

새 문서:
260618_04_Anima Spire 034A_Codex_구현지시_UI탭구조MVP_v01
```

같은 날짜에 이미 작성된 문서가 있는 경우 다음 번호를 사용한다.

Drive 검색이 가능하지만 결과가 없으면 해당 날짜의 첫 문서로 보고 `01`을 사용할 수 있다.

Drive 검색이 불가능하거나, 검색 결과가 불명확하거나, 사용자가 이미 Drive에 수동으로 작성한 문서가 있을 가능성이 있으면 AI는 일련번호를 임의로 확정하지 않는다.

이 경우 AI는 사용자에게 해당 날짜의 최신 일련번호 확인을 요청한 뒤 파일명을 제안한다.

파일명을 임시로 제안해야 하는 경우에는 `NN`을 그대로 두거나, `일련번호 확인 필요`로 표시한다.

예시는 다음과 같다.

```text id="4ekt15"
260618_NN_Anima Spire 034A_Gemini_검토세트_UI탭구조설계검토_v01
```

또는 다음과 같이 표시할 수 있다.

```text id="r3o0yr"
260618_일련번호확인필요_Anima Spire 034A_Gemini_검토세트_UI탭구조설계검토_v01
```

AI는 Drive 확인 없이 해당 날짜의 일련번호를 확정값처럼 작성하지 않는다.

### 5.6 버전번호 기준

Drive 저장문서 버전은 `v01`, `v02`, `v03` 형식으로 표기한다.

소스 기준문서 내부 버전도 `v01`, `v02`, `v03` 형식으로 표기한다.

정규 기획문서 중 전체 기획서와 카테고리별 기획서는 필요에 따라 `rXXX.XX` 계열을 사용할 수 있다.

MVP 구현로드맵, 완료반영판, 인수인계, 폐기결정, AI협업문서는 원칙적으로 `vXX` 계열을 사용한다.

### 5.7 보완루프 문서 버전 기준

보완루프 이후 새로 생성되는 검토세트, 검색세트, 구현지시, 검토지시는 신규 문서로 생성한다.

보완루프 문서의 작업차수는 `v02`부터 시작한다.

재보완 시 `v03`, `v04` 형식으로 증가한다.

예시는 다음과 같다.

```text
260618_08_Anima Spire 034A_Gemini_검토세트_UI탭구조회귀위험_보완설계검토_v02
260618_09_Anima Spire 034A_Codex_구현지시_UI탭구조회귀위험_보완구현_v02
```

작업차수는 문서 수정 버전이 아니라 동일 작업단계 내 보완 차수를 의미한다.

### 5.8 폐기결정 문서 파일명

폐기결정 문서는 AI명 없이 작성한다.

기본 형식은 다음과 같다.

```text
YYMMDD_NN_Anima Spire {단계번호}_폐기결정_{문서제목}_vXX
```

예시는 다음과 같다.

```text
260618_10_Anima Spire 034A_폐기결정_UI탭구조접근방식폐기_v01
```

### 5.9 빌드 파일명 기준

Android 테스트빌드, 실기기 검증빌드, 릴리즈 후보빌드 등 빌드 산출물의 파일명은 다음 구조를 따른다.

```text
AnimaSpire_STEP{MVP단계5자리}_{작업내용}_{빌드차수2자리}
```

예시는 다음과 같다.

```text
AnimaSpire_STEP033J0_Android실기기검증및마무리_02
```

각 요소의 의미는 다음과 같다.

| 요소               | 의미                         |
| ---------------- | -------------------------- |
| `AnimaSpire`     | 프로젝트명                      |
| `STEP{MVP단계5자리}` | 빌드가 생성된 MVP 단계 식별자         |
| `{작업내용}`         | 해당 빌드의 목적 또는 작업 내용을 압축한 설명 |
| `{빌드차수2자리}`      | 해당 MVP 단계에서 생성된 빌드 차수      |

`STEP` 뒤의 MVP 단계 식별자는 5자리로 작성한다.

MVP 단계 식별자는 다음 형식을 따른다.

```text
숫자 3자리 + 영문 1자리 + 숫자 1자리
```

예시는 다음과 같다.

```text
033J0
033J3
034A0
```

`033J0`은 `033J` 단계에서 생성된 기본 빌드 또는 `033J-0`에 해당하는 빌드를 의미한다.

`033J3`은 `033J-3` 단계에서 생성된 빌드를 의미한다.

작업내용은 해당 빌드의 목적을 짧게 설명한다.

작업내용에는 필요한 경우 영문과 한글을 함께 사용할 수 있다.

빌드차수는 해당 MVP 단계에서 생성된 빌드 순서를 의미한다.

첫 번째 빌드는 `_01`, 두 번째 빌드는 `_02`, 세 번째 빌드는 `_03` 형식으로 작성한다.

예시는 다음과 같다.

```text
AnimaSpire_STEP033J0_Android실기기검증및마무리_01
AnimaSpire_STEP033J0_Android실기기검증및마무리_02
AnimaSpire_STEP033J3_UI보완빌드_01
AnimaSpire_STEP034A0_탭구조초기검증_01
```

빌드 파일명은 빌드 산출물 자체의 이름에 적용한다.

빌드·릴리즈 기록 문서의 파일명은 Drive 저장문서 파일명 기준을 따른다.

Codex와 ClaudeCode는 Android APK 또는 AAB 빌드를 수행하지 않는다.

Android 빌드 수행 주체와 Android 빌드 금지 기준은 `IMPLEMENTATION_GUARDRAILS.md`를 따른다.

## 6. Drive 폴더 저장 기준

### 6.1 기본 원칙

Drive는 기획문서, MVP 로드맵, AI협업문서, 인수인계, 테스트·이슈기록, 빌드·릴리즈 자료, 운영기록, 임시문서, 아카이브 문서를 보관하는 공간으로 운영한다.

소스 기준문서 4종은 GitHub 소스에 저장한다.

Drive에는 기준문서 개편 기안문, 기준문서 검토문, 기준문서 최종검토 결과 등 협업 산출물을 저장할 수 있다.

### 6.2 개편 후 Drive 폴더 구조

Anima Spire Drive 폴더는 다음 구조를 기준으로 운영한다.

```text
01_기획문서
  01-01_전체기획
  01-02_카테고리별기획

02_MVP로드맵

03_AI협업문서
  03-01_기안문
  03-02_기획단계협업
  03-03_작업단계협업

04_인수인계

05_테스트_이슈기록

06_빌드_릴리즈

90_운영_기준문서_기록

98_임시문서

99_아카이브
```

사용자는 본 문서 기준에 맞춰 Drive 폴더 구조를 직접 정비한다.

AI는 문서 작성 시 위 구조를 기준으로 추천 저장 위치를 안내한다.

### 6.3 단계별 저장 기준

Drive 저장 위치는 문서 유형만으로 판단하지 않고, 해당 문서가 발생한 작업 단계와 목적을 기준으로 판단한다.

검토세트와 검색세트는 기획단계와 작업단계 양쪽에서 모두 발생할 수 있으므로, 문서 유형만으로 저장 위치를 고정하지 않는다.

같은 Gemini 검토세트라도 구현 전 기획, 방향, 구조, 범위, 로드맵 판단을 위한 문서이면 `03_AI협업문서/03-02_기획단계협업`에 저장한다.

같은 Gemini 검토세트라도 특정 MVP 또는 세부 작업단계의 구현, 보완, 코드검토, 완료보고, 테스트 대응과 연결된 문서이면 `03_AI협업문서/03-03_작업단계협업`에 저장한다.

Perplexity 검색세트도 동일하게 적용한다.

외부 정책, 공식문서, 기술 이슈 검색이 기획 판단을 위한 것이면 `03_AI협업문서/03-02_기획단계협업`에 저장한다.

특정 작업의 구현, 테스트, 회귀 위험, Android 빌드, Unity 동작 확인을 위한 검색이면 `03_AI협업문서/03-03_작업단계협업`에 저장한다.

기준문서 개편, 운영 기준, AI 협업 구조, 최종검토 결과 등 프로젝트 운영 성격의 문서는 `90_운영_기준문서_기록`에 저장한다.

기존 AI 협업문서는 소급 분류하지 않는다.

기존 작업지시문, 검토요청문, 검토답변, 구현 완료보고, 보완지시문, 기존 작업 관련 AI 협업문서는 `03_AI협업문서/03-03_작업단계협업`에 일괄 보관할 수 있다.

### 6.4 세트문서 저장 방식

세트문서는 발신문서와 회신문서를 하나의 Drive 문서 안에서 관리한다.

GPT는 발신문서를 문서블록으로 작성한다.

사용자는 발신문서를 Drive 문서로 저장한 뒤 다른 AI에게 전달한다.

다른 AI의 회신은 부연설명 없이 채팅에 직접 출력하도록 요청한다.

사용자는 Drive 문서 안에 구분선을 삽입한 뒤 회신 전문을 붙여넣는다.

권장 구분선은 다음과 같다.

```text
---

# [회신 원문]
```

필요한 경우 GPT는 회신 이후 다음 구분을 추가로 작성할 수 있다.

1. GPT 해석
2. 반영 여부
3. 사용자 결정 필요 항목
4. 보류 또는 제외 항목

### 6.5 저장 안내 방식

AI가 Drive 저장문서를 작성할 때에는 다음을 함께 안내한다.

1. 추천 파일명
2. 추천 저장 폴더
3. 문서 성격
4. 후속 검토 필요 여부

추천 저장 폴더는 문서 유형만으로 결정하지 않고, 해당 문서가 기획단계 협업문서인지 작업단계 협업문서인지 함께 판단한다.

예시는 다음과 같다.

```text
추천 파일명:
260618_06_Anima Spire 034A_Codex_구현지시_UI탭구조MVP_v01

추천 저장 위치:
03_AI협업문서/03-03_작업단계협업

문서 성격:
Codex 전달용 구현지시문

후속 검토:
사용자 검토 후 Codex에 전달
```

기획단계 협업문서 예시는 다음과 같다.

```text
추천 파일명:
260618_05_Anima Spire 034A_Gemini_검토세트_UI탭구조설계검토_v01

추천 저장 위치:
03_AI협업문서/03-02_기획단계협업

문서 성격:
구현 전 설계안 검토세트

후속 검토:
사용자 검토 후 Gemini에 전달
```

작업단계 협업문서 예시는 다음과 같다.

```text
추천 파일명:
260618_07_Anima Spire 034A_ClaudeCode_검토지시_UI탭구조회귀검토_v01

추천 저장 위치:
03_AI협업문서/03-03_작업단계협업

문서 성격:
구현 결과 코드검토지시문

후속 검토:
사용자 검토 후 ClaudeCode에 전달
```

## 7. 문서 유형별 출력 기준

### 7.1 검토세트

검토세트는 Gemini, ClaudeCode 등에게 설계안, 방향, 구조, 리스크를 검토받기 위한 문서다.

검토세트는 원칙적으로 다음 구조를 따른다.

```markdown
# [검토요청-Gemini]

## 1. 검토 목적

## 2. 현재 상황

## 3. 관련 기준문서 및 참고자료

## 4. 구현 가드레일 대조 필요 여부

## 5. 검토 대상

## 6. 제외 범위

## 7. 주요 쟁점

## 8. 질문 목록

## 9. 답변 요청 형식
```

검토세트는 검토 대상이 무엇인지 명확히 해야 한다.

검토 대상이 구현 구조, 회귀 위험, SaveData, Scene, UI, Android, Combat, Equipment, Legacy code와 관련되는 경우에는 `IMPLEMENTATION_GUARDRAILS.md`의 관련 장을 검토세트에 포함하거나 요약하여 제시한다.

이 경우 Gemini는 설계안의 기술적 타당성뿐 아니라 구현 가드레일 위반 가능성도 함께 검토하도록 요청받는다.

설계안검토는 작업지시문 작성 전 설계안, 원인 가설, 보완 방향, 리스크를 검토하는 문서다.

지시문검토는 작업지시문 자체를 검토하는 행위이며, 정식 검토세트 문서로 저장하지 않는 것을 원칙으로 한다.

### 7.2 검색세트

검색세트는 Perplexity 또는 외부 검색 AI에게 공식 문서, 최신 정보, 외부 이슈 사례를 확인하도록 요청하는 문서다.

검색세트는 원칙적으로 다음 구조를 따른다.

```markdown
# [검색요청-Perplexity]

## 1. 검색 목적

## 2. 현재 상황

## 3. 확인해야 할 항목

## 4. 우선 검색 대상

## 5. 제외할 정보

## 6. 답변 요청 형식

## 7. 출처 기준
```

검색세트에는 출처 기반 답변을 요구한다.

최신 정보가 필요한 경우 기준 날짜를 명시한다.

확인 불가한 항목은 확인 불가라고 답하도록 요청한다.

### 7.3 Claude 기안문 요청문

Claude 기안문 요청문은 기획, 로드맵, 방향, 기준문서 개편 등 Claude가 담당하는 상위 판단을 요청할 때 작성한다.

요청문은 원칙적으로 다음 구조를 따른다.

```markdown
# [기안요청-Claude]

## 1. 요청 목적

## 2. 현재 상황

## 3. 제공된 기준문서 및 참고자료

## 4. 사용자 의도

## 5. 검토 또는 기안 요청 범위

## 6. 제외 범위

## 7. 답변 요청 형식

## 8. 주의사항
```

Claude에게 구현지시문 작성, Git 명령어 안내, 완료보고 검토, Unity 테스트 체크리스트 작성, Drive 파일명·저장 위치 최종결정을 요청하지 않는다.

단, 사용자가 명시적으로 Claude에게 역할 범위 밖 작업을 요청한 경우에는 `02_AI_WORKFLOW_RULES.md`의 사용자 명시 요청 예외 규정을 따른다.

### 7.4 구현지시문

구현지시문은 Codex 또는 ClaudeCode가 실제 구현을 수행할 수 있도록 작성하는 자급자족형 지시문이다.

구현자는 Drive 문서나 Claude 대화 내용을 직접 확인하지 못할 수 있으므로, 구현지시문은 항상 자급자족형으로 작성한다.

구현지시문은 원칙적으로 다음 구조를 따른다.

```markdown
# [작업지시-GPT]

## 1. 작업 목적

## 2. 현재 상태

## 3. 작업 범위

## 4. 수정 대상

## 5. 수정 금지 범위

## 6. 구현 요구사항

## 7. 구현 가드레일

## 8. 테스트 또는 확인 요청

## 9. 완료보고 요구사항

## 10. 금지사항
```

구현지시문에는 다음을 반드시 포함한다.

1. 작업 대상 단계
2. 작업 목적
3. 수정할 파일 또는 영역
4. 수정하지 말아야 할 파일 또는 영역
5. 구현 요구사항
6. 금지사항
7. 완료보고에 포함해야 할 항목
8. Git write 금지
9. Android APK/AAB 빌드 금지
10. `IMPLEMENTATION_GUARDRAILS.md` 준수

Claude 원문 전체를 구현지시문에 그대로 붙여넣지 않는다.

Claude 기안 또는 검토결과는 GPT가 구현자에게 필요한 내용만 추출하여 반영한다.

### 7.5 검토지시문

검토지시문은 ClaudeCode 또는 코드검토자에게 실제 코드 기준 확인을 요청하는 문서다.

검토지시문은 원칙적으로 다음 구조를 따른다.

```markdown
# [검토지시-GPT]

## 1. 검토 목적

## 2. 검토 대상

## 3. 확인할 파일 또는 영역

## 4. 확인하지 말아야 할 범위

## 5. 검토 기준

## 6. 회귀 위험 확인 항목

## 7. 완료보고 요구사항

## 8. 금지사항
```

파일 수정 없는 검토를 요청하는 경우 다음 문구를 포함한다.

```text
이번 작업은 파일 수정 없는 코드검토입니다.
코드를 수정하지 말고, 확인 결과와 위험 요소만 보고해 주세요.
```

파일 수정을 동반하는 경우에는 검토지시문이 아니라 구현지시문으로 작성한다.

### 7.6 완료보고 검토 출력

구현자의 완료보고를 검토할 때 GPT는 원칙적으로 다음 형식을 따른다.

```text
[상황 판단]
완료보고 기준으로 작업이 어떤 상태인지 판단한다.

[확인된 사항]
- 지시문 이행 여부
- 수정 파일
- 테스트 수행 여부
- 금지사항 위반 여부

[주의 또는 확인 필요]
- 확인이 필요한 항목
- 사용자 테스트가 필요한 항목
- 회귀 가능성

[다음 단계]
- 테스트 진행
- 보완루프
- Claude 방향점검
- 커밋대상 확인
- 보류
```

완료보고만으로 정상 확정하지 않는다.

사용자 Unity 또는 Android 테스트가 필요한 항목은 별도로 안내한다.

### 7.7 테스트 안내 출력

테스트 안내는 작업지시문, 완료보고, 구현 가드레일, 회귀 위험을 기준으로 작성한다.

테스트 안내는 원칙적으로 다음 구조를 따른다.

```text
[테스트 목적]

[Unity Editor 확인 항목]

[Android 실기기 확인 항목]

[Console 또는 로그 확인 항목]

[정상 기준]

[문제 발생 시 보고할 내용]
```

Android A영역, Safe Area, 상태바, 시스템 제스처, 카메라홀 관련 검증은 Android 실기기 확인 항목으로 분리한다.

### 7.8 폐기결정 문서

폐기결정 문서는 특정 작업 결과 또는 작업단계 접근을 폐기하기로 사용자 결정이 내려진 경우 작성한다.

폐기결정 문서는 원칙적으로 다음 구조를 따른다.

```markdown
# [폐기결정]

## 1. 폐기 대상

## 2. 폐기 결정 사유

## 3. 폐기 범위

## 4. 유지할 내용

## 5. 되돌릴 내용

## 6. 후속 작업 방향

## 7. 재사용 금지 또는 참고 가능 항목

## 8. 결정 이력
```

폐기결정은 AI가 단독으로 확정하지 않는다.

사용자 결정 후 작성한다.

### 7.9 인수인계 문서

인수인계 문서는 새 채팅, 다음 작업자, 후속 작업단계 진입을 위해 현재 상태를 정리하는 문서다.

인수인계 문서는 원칙적으로 다음 구조를 따른다.

```markdown
# [인수인계]

## 1. 문서 목적

## 2. 현재 완료 상태

## 3. Git 상태

## 4. Drive 문서 상태

## 5. 사용자 결정사항

## 6. 남은 확인 필요사항

## 7. 다음 작업 후보

## 8. 새 채팅 시작 프롬프트
```

확인하지 못한 Git 상태나 Drive 저장 상태를 확정 사실처럼 쓰지 않는다.

### 7.10 MVP 구현로드맵

MVP 구현로드맵은 하나의 MVP 작업단위를 구현 가능한 세부단계로 나누는 문서다.

MVP 구현로드맵은 원칙적으로 다음 구조를 따른다.

```markdown
# [MVP 구현로드맵]

## 1. 문서 목적

## 2. MVP 목표

## 3. 적용 범위

## 4. 제외 범위

## 5. 현재 기준문서 및 참고자료

## 6. 세부단계 구성

## 8. 테스트 기준

## 9. 리스크 및 보류 사항

## 10. 다음 단계
```

MVP 구현로드맵은 Claude 최종검토 적용 대상이다.

### 7.11 MVP 구현로드맵 완료반영판

MVP 구현로드맵 완료반영판은 해당 MVP 완료 후 실제 구현 결과, 테스트 결과, 남은 보류사항을 반영하는 문서다.

```markdown
# [MVP 구현로드맵 완료반영판]

## 1. 문서 목적

## 7. Git 및 브랜치 정리 상태

## 8. 다음 MVP 진입 전 확인사항
```

MVP 구현로드맵 완료반영판은 Claude 최종검토 적용 대상이다.

7. MVP 구현로드맵
8. MVP 구현로드맵 완료반영판

단일문서는 요청문과 답변서를 나누지 않는다.

### 8.2 세트문서

세트문서는 요청문과 답변 또는 결과를 함께 관리하는 문서다.

예시는 다음과 같다.

1. Gemini 검토세트
2. Perplexity 검색세트
3. Claude 방향점검 세트
4. Claude 기안문 검토세트

세트문서는 원칙적으로 다음 구조를 가진다.

```markdown
# [세트문서 제목]

## 1. 요청문

---

# [회신 원문]

## 1. 회신 내용

---

# [GPT 해석]

## 1. 핵심 판단

## 2. 반영 여부

## 3. 사용자 결정 필요 항목
```

사용자는 회신 전문을 직접 Drive 문서에 붙여넣는다.

GPT는 회신을 요약만으로 대체하지 않는다.

추후 검토가 가능하도록 회신 전문을 보존한다.

### 8.3 지시문검토 문서화 기준

지시문검토는 정식 Drive 저장문서로 만들지 않는 것을 원칙으로 한다.

따라서 파일명 생성 규격을 적용하지 않는다.

지시문검토는 기본적으로 채팅 내 검토로 처리한다.

다만 지시문검토 결과를 반영해 추가 지시문구를 작성하는 경우, 해당 추가 지시문구는 후속 작업지시문 보완 또는 보완루프 문서에 반영할 수 있다.

---

## 9. 정규문서 출력 기준

### 9.1 정규문서 공통 구조

정규문서는 원칙적으로 다음 구조를 가진다.

```markdown
# [정규문서 제목]

## 1. 문서 목적

## 2. 적용 범위

## 3. 기준 문서 및 참고자료

## 4. 본문

## 5. 보류 또는 제외 사항

## 6. 사용자 결정사항

## 7. 후속 작업

## 8. 변경 이력
```

문서 성격에 따라 일부 항목을 생략하거나 다른 제목으로 바꿀 수 있다.

단, 사용자 결정사항, 보류사항, 제외사항이 있는 경우에는 본문과 분리하여 적는다.

### 9.2 전체 기획서와 카테고리별 기획서

전체 기획서와 카테고리별 기획서는 기획 기준을 관리하는 정규문서다.

이 문서들은 필요에 따라 `rXXX.XX` 계열을 사용할 수 있다.

문서에는 다음을 명확히 구분한다.

1. 확정된 기획
2. 보류된 기획
3. 제외된 기획
4. 향후 검토 후보
5. 현재 구현 상태와 최종 서비스 비전의 차이

### 9.3 기준문서 개정 산출물

기준문서 개정 산출물은 다음을 구분하여 저장한다.

1. Claude 기안문
2. GPT 문서화 초안
3. Claude 최종검토 결과
4. 사용자 확정본
5. 소스 md 반영본

기준문서 최종본에는 개편 사유를 장황하게 쓰지 않는다.

개편 경위는 변경 이력에 간단히 기록한다.

## 10. 완료보고 요구사항 출력 기준

### 10.1 구현자 완료보고 요구사항

구현지시문에는 완료보고 요구사항을 반드시 포함한다.

완료보고에는 다음 항목을 요구한다.

1. 수정한 파일 목록
2. 주요 변경 내용
3. 지시문 이행 여부
4. 수정하지 않은 금지 영역
5. 테스트 수행 여부
6. 테스트하지 못한 항목과 사유
7. 남은 위험 또는 후속 필요사항
8. Git write 작업을 하지 않았다는 명시
9. Android APK/AAB 빌드를 하지 않았다는 명시

### 10.2 검토자 완료보고 요구사항

검토지시문에는 다음 완료보고 항목을 요구한다.

1. 검토한 파일 또는 영역
2. 지시문 기준 충족 여부
3. 발견된 문제
4. 회귀 위험
5. 수정 없이 검토만 수행했는지 여부
6. 추가 확인 필요사항

---

## 11. 기준문서 개정 출력 기준

### 11.1 기준문서 초안 작성 방식

기준문서 초안은 문서별로 작성한다.

한 응답에서 4개 기준문서 전체를 모두 완성하려고 하지 않는다.

기준문서는 원칙적으로 다음 순서로 작성한다.

1. `02_AI_WORKFLOW_RULES.md`
2. `03_AI_OUTPUT_RULES.md`
3. `IMPLEMENTATION_GUARDRAILS.md`
4. `01_AI_ROUTER.md`

단, 사용자가 요청하면 임시 라우터 초안을 먼저 작성할 수 있다.

### 11.2 사용자 검토 방식

기준문서 초안을 작성한 뒤 AI는 다음을 사용자에게 확인한다.

1. 수정이 필요한 항목
2. 사용자 결정이 필요한 항목
3. 삭제 또는 축약할 항목
4. 다른 문서로 이관할 항목
5. Claude 최종검토가 필요한 시점

### 11.3 최종본 작성 방식

최종본에서는 초안 작성 경위, 기존 문서체계의 문제점, 개편 배경 설명을 본문에 길게 쓰지 않는다.

최종본은 앞으로 적용할 기준만 명확히 적는다.

개편 경위는 변경 이력에 간단히 기록한다.

### 11.4 기준문서 내부 버전 표기

소스 기준문서의 파일명에는 버전을 붙이지 않는다.

문서 내부 첫 줄 제목에는 `_vXX` 형식의 버전을 붙인다.

변경 이력은 `v01`, `v02` 형식으로 작성한다.

예시는 다음과 같다.

```markdown
# AI Output Rules - Anima Spire 출력기준_v02

## 12. 변경 이력

v01
- 최초 작성.

v02
- 사용자 결정사항 반영.
```

---

## 12. 변경 이력

v01

* `GPT_WORK_RULES.md`, `CLAUDE_WORK_RULES.md`, `PROJECT_RULES.md`, `AI_COLLABORATION_BOOTSTRAP_PACKAGE.md`의 출력기준 관련 내용을 바탕으로 출력기준 초안 작성.
* 판단기준은 `02_AI_WORKFLOW_RULES.md`로 분리.
* 구현자 가드레일은 `IMPLEMENTATION_GUARDRAILS.md`로 분리.
* 파일명, 문서 템플릿, Drive 저장 위치, 코드블록, 터미널 명령어, 문서 유형별 출력 기준을 본 문서에 통합.

v02

* 사용자 결정사항 반영.
* 파일명을 생성하는 Drive 저장문서에만 강제 출력규격을 적용하는 기준 추가.
* 모든 파일명은 코드블록으로 작성하는 기준 확정.
* 단독문서와 발신문서는 마크다운 렌더링 문서블록으로 작성하는 기준 확정.
* 세트문서 회신은 부연설명 없이 채팅 직접출력하고, 지정된 헤딩 규격을 따르도록 정리.
* 발신문서 안에 회신 출력규격을 반드시 명시하는 기준 추가.
* 지시문검토는 정식 Drive 저장문서로 만들지 않는 원칙 유지.
* Drive 폴더 구조를 개편 후 기준으로 정리.
* 세트문서 회신 전문은 사용자가 Drive 문서 내 구분선 아래에 직접 붙여넣는 방식으로 확정.
* 소스 기준문서 내부 버전 표기 기준을 `_vXX`, 변경 이력 표기를 `vXX` 형식으로 정리.
* Gemini 검토세트에 구현 가드레일 대조 필요 여부를 포함할 수 있도록 보완.
* 빌드 산출물 파일명 기준을 `AnimaSpire_STEP{MVP단계5자리}_{작업내용}_{빌드차수2자리}` 형식으로 추가.
* Drive 폴더 저장 기준을 문서 유형 중심에서 기획단계협업·작업단계협업 중심으로 정리하고, 기존 AI 협업문서는 소급 분류하지 않는 기준을 추가.
