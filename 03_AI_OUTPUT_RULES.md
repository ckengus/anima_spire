# AI Output Rules - Anima Spire 출력기준_v03

## 1. 공통 출력 원칙

### 1.1 문서 목적

이 문서는 Anima Spire 프로젝트에서 GPT, Claude, Gemini, Perplexity, Codex, ClaudeCode 등 협업 AI가 산출물을 작성할 때 적용하는 출력기준 문서다.

이 문서는 판단기준 문서가 아니다.

현재 요청의 작업 유형, 담당 AI, 사용자 결정 필요 여부, 보완·폐기·방향점검 전환 여부는 `02_AI_WORKFLOW_RULES.md`를 따른다.

현재 요청이 파일명 생성 대상 프로젝트 저장문서 작성 요청인지, 일반 채팅 응답 요청인지는 `02_AI_WORKFLOW_RULES.md`의 문서작성 요청 판단 기준을 따른다.

이 문서는 문서작성 요청으로 판단된 이후 다음을 정한다.

1. 어떤 형식으로 응답할지
2. 어떤 파일명을 사용할지
3. 어떤 문서블록·코드블록·채팅 직접출력 기준을 적용할지
4. 어떤 문서 템플릿 또는 문서 구조를 사용할지
5. Drive 저장 위치를 어떻게 안내할지
6. AI협업문서와 개발문서를 어떤 구조로 출력할지
7. 발신문서에 회신규격을 어떻게 명시할지

구현자 또는 코드검토자 가드레일은 `IMPLEMENTATION_GUARDRAILS.md`를 따른다.

### 1.2 적용 대상

이 문서는 `02_AI_WORKFLOW_RULES.md` 기준에 따라 문서작성 요청으로 판단된 산출물에 적용한다.

이 문서는 다음 경우에 적용한다.

1. 파일명을 생성하는 Drive 저장문서를 작성하는 경우
2. 사용자가 저장하거나 다른 AI에게 전달할 정식 문서 본문을 작성하는 경우
3. Claude, Gemini, Perplexity, Codex, ClaudeCode에 보낼 발신문서를 문서로 작성하는 경우
4. AI 회신 형식을 발신문서 안에 명시해야 하는 경우
5. 코드블록, 문서블록, 채팅 직접출력 방식을 구분해야 하는 경우
6. Drive 저장 위치를 안내해야 하는 경우
7. 정규문서, 인수인계, 폐기결정, MVP 구현로드맵, MVP 구현로드맵 완료반영판을 문서로 작성하는 경우
8. 테스트·이슈기록, 빌드·릴리즈 기록을 문서로 작성하는 경우
9. 터미널 명령어 또는 Git 후속 안내를 출력하는 경우
10. 소스 기준문서 4종의 개정본문 또는 교체본문을 출력하는 경우

문서작성으로 판단된 경우에는 파일명, 문서블록, 헤딩, 회신규격, 저장 위치 안내 등 이 문서의 출력규격을 적용한다.

### 1.3 비적용 대상

명시적 문서작성 요청이 없는 일반 응답, 간단한 판단, 짧은 문구 수정, 단순 질의응답은 이 문서의 저장문서 출력규격을 강제 적용하지 않는다.

사용자가 명시적으로 문서작성을 요청하지 않은 검토, 설명, 요약, 정리, 의견 요청, 전달문 작성 요청은 일반 채팅 응답으로 처리한다.

다른 AI에게 전달할 내용 작성 요청이라도, 사용자가 명시적으로 문서 작성 또는 `[문서]` 트리거를 요청하지 않은 경우에는 이 문서의 발신문서 출력규격을 자동 적용하지 않는다.

수신 대상이 Gemini, Claude, Perplexity, Codex, ClaudeCode라는 이유만으로 발신문서 규격을 자동 적용하지 않는다.

단, 일반 응답이라도 사용자가 특정 출력형식, 문서블록, 코드블록, 파일명, 저장 위치를 명시적으로 요청한 경우에는 이 문서의 해당 장을 따른다.

### 1.4 언어와 말투

Anima Spire 프로젝트 응답은 원칙적으로 한국어 존대말로 작성한다.

다만 `IMPLEMENTATION_GUARDRAILS.md`, Codex 구현지시문, ClaudeCode 검토지시문 등 구현자용 영문 가드레일 또는 영문 지시가 필요한 문서는 영어로 작성할 수 있다.

응답은 실무 작업 흐름에 맞게 명확하고 차분하게 작성한다.

줄임말, 이모티콘, 과한 감탄 표현은 사용하지 않는다.

### 1.5 확인 불가 항목 출력 기준

확인하지 않은 항목은 확정 표현으로 쓰지 않는다.

확인할 수 없는 항목은 다음 중 하나로 분리한다.

1. 확인 완료
2. 확인 필요
3. 사용자 제공 필요
4. 보류
5. 추정 금지

AI는 확인 불가 항목을 문서 본문에 확정 사실처럼 포함하지 않는다.

### 1.6 제안과 결정의 구분

문서나 응답에서 다음 상태를 구분하여 표현한다.

1. 결정
2. 제안
3. 보류
4. 제외
5. 확인 필요

사용자가 명시적으로 확정하지 않은 항목은 “결정”으로 쓰지 않는다.

## 2. 일반 응답 출력 형식

### 2.1 판단·제안 응답 기본 형식

AI가 판단, 방향 제안, 작업 흐름 전환, 사용자 결정 요청을 포함하는 응답을 할 때에는 원칙적으로 다음 형식을 따른다.

```text
[상황 판단]
현재 상황이 어떤 기준에 해당하는지 간단히 정리한다.

[제안 내용]
제안이 하나면 단일안으로 제시한다.
제안이 둘 이상이면 A안 / B안 / C안으로 병렬 제시한다.
각 안의 핵심 장단점을 짧게 정리한다.

[결정 필요 항목]
□ 항목 1
□ 항목 2
□ 항목 3

[비고]
추가 확인 필요사항, 판단하지 못한 부분, 충돌 가능성만 짧게 적는다.
없으면 생략한다.
```

### 2.2 기본 형식을 생략할 수 있는 경우

다음 경우에는 2.1 형식을 생략할 수 있다.

1. 사용자가 단일 사실만 묻는 경우
2. 사용자가 예/아니오 답변만 요청한 경우
3. 사용자가 명시적으로 본문만 요청한 경우
4. 간단한 문구 수정 요청인 경우
5. 이미 확정된 내용을 단순 정리하는 경우
6. 파일명을 생성하지 않는 자유 응답인 경우

### 2.3 기본 형식을 반드시 적용하는 경우

다음 경우에는 2.1 형식을 적용한다.

1. 판단이 포함된 응답
2. 방향 제안이 포함된 응답
3. 작업 흐름 전환이 포함된 응답
4. Claude 방향점검 트리거 발동 시
5. 폐기, 보완, 로드맵 재설정 중 선택을 제안하는 경우
6. 기준문서 개정 방향을 제안하는 경우
7. 사용자 결정이 필요한 항목이 있는 경우

### 2.4 다음 단계 안내 형식

모든 응답 말미에 다음 단계 블록을 기계적으로 반복하지 않는다.

다음 단계 안내는 다음 기준을 따른다.

1. 새 다음 단계가 생겼을 때 1회 안내한다.
2. 사용자가 즉시 진행하지 않고 다른 대화가 이어지면 반복하지 않는다.
3. 보류된 다음 단계가 2개 이상이면 모두 제시하고 우선순위 결정을 요청한다.
4. 현재 주제가 마무리되었다고 판단되면 추가 논의 또는 다음 단계 안내 여부를 묻는다.

다음 단계 안내 형식은 다음을 따른다.

```text
**다음 단계**
담당: GPT / Claude / Gemini / Perplexity / Codex / ClaudeCode / 사용자
할 일: 한 줄로 작성
필요한 것: 필요한 경우에만 작성
```

다음 단계 안내는 사용자의 명시적 결정을 대체하지 않는다.

## 3. 문서블록·코드블록·채팅 직접출력 기준

### 3.1 적용 기준

이 장은 파일명을 생성하는 Drive 저장문서, AI 간 발신문서, 정식 문서본문, 기준문서 본문, 구현지시문, 검토지시문, 검색요청문, 검토요청문 등 문서 산출물을 출력할 때 적용한다.

이 장은 `02_AI_WORKFLOW_RULES.md` 기준에 따라 문서작성 요청으로 판단된 경우에 적용한다.

파일명을 생성하지 않는 일반 응답, 간단한 판단, 짧은 문구 수정, 단순 질의응답, 명시적 문서작성 요청이 없는 전달문 작성 요청에는 이 장의 저장문서 출력규격을 강제하지 않는다.

다만 사용자가 문서블록, 코드블록, 파일명, 저장 위치 등 특정 출력형식을 명시적으로 요청한 경우에는 해당 출력형식을 따른다.

### 3.2 파일명 출력 기준

파일명을 생성하는 모든 문서의 파일명은 코드블록으로 작성한다.

파일명은 일반 문장 안에만 흘려 쓰지 않고, 사용자가 그대로 복사할 수 있도록 별도 코드블록으로 제공한다.

예시는 다음과 같다.

```text
260618_01_Anima Spire 운영_GPT_검토요청문_034진입전기준문서정리_v01
```

여러 파일명을 제시하는 경우에도 코드블록 안에 한 줄씩 작성한다.

```text
01_AI_ROUTER.md
02_AI_WORKFLOW_RULES.md
03_AI_OUTPUT_RULES.md
IMPLEMENTATION_GUARDRAILS.md
```

파일명 코드블록은 문서본문과 분리하여 출력한다.

### 3.3 문서 본문 출력 기준

파일명을 생성하는 단독문서와 발신문서의 본문은 마크다운 렌더링이 적용되는 하나의 문서블록으로 작성한다.

Drive에 저장할 문서본문 전체를 `markdown` 코드블록 안에 넣지 않는다.

문서본문은 사용자가 바로 읽고 검토할 수 있도록 마크다운 헤딩, 표, 목록이 렌더링되는 형태로 작성한다.

문서본문에 포함되는 터미널 명령어, 파일명 예시, 코드 예시는 필요한 경우 문서본문 내부에서 코드블록으로 작성할 수 있다.

문서작성 모드 안내, 저장 위치 안내, 후속 검토 필요 여부 등 문서본문에 포함되지 않는 안내는 문서블록 밖 일반 채팅 영역에 작성한다.

### 3.4 단독문서 출력 기준

다음 문서는 파일명 코드블록과 하나의 문서블록 본문으로 작성한다.

1. 정규문서
2. 인수인계 문서
3. 폐기결정 문서
4. MVP 구현로드맵
5. MVP 구현로드맵 완료반영판
6. 기준문서 본문
7. 테스트·이슈기록
8. 빌드·릴리즈 기록

단독문서의 제목은 문서 성격을 기준으로 작성하되, AI 작성자 표기가 필요한 경우 작성자 기준으로 표기한다.

### 3.5 발신문서 출력 기준

다음 문서는 파일명 코드블록과 하나의 문서블록 본문으로 작성한다.

1. Claude 기안 요청문
2. Claude 방향점검 요청문
3. Gemini 검토요청문
4. Perplexity 검색요청문
5. Codex 구현지시문
6. ClaudeCode 검토지시문
7. ClaudeCode 구현지시문

발신문서는 사용자가 다른 AI에게 전달하기 위해 GPT 또는 Claude가 작성하는 요청문이다.

발신문서의 제목은 수신 AI가 아니라 문서 작성자 기준으로 작성한다.

예시는 다음과 같다.

```text
# [검토요청-GPT]
# [검색요청-GPT]
# [기안요청-GPT]
# [작업지시-GPT]
# [검토지시-GPT]
```

발신문서에는 회신 출력규격을 반드시 포함한다.

Gemini, Perplexity, ClaudeCode, Codex 등 외부 AI 또는 구현자가 회신 형식을 지키지 않을 가능성이 있으므로, GPT 또는 Claude가 작성하는 발신문서에는 답변 제목, 헤딩 규격, 부연설명 금지, 확인 불가 처리 기준을 명시한다.

다만 사용자가 명시적으로 문서작성을 요청하지 않은 일반 전달문 작성 요청에는 발신문서 출력규격을 자동 적용하지 않는다.

### 3.6 회신문서 출력 기준

AI 간 세트문서의 회신은 부연설명 없이 본문 내용만 채팅에 직접 출력하도록 요청한다.

다음은 회신문서에 해당한다.

1. Claude 기안 답변
2. Claude 방향점검 답변
3. Gemini 검토답변
4. Perplexity 검색결과
5. Codex 완료보고
6. ClaudeCode 검토답변
7. ClaudeCode 구현 완료보고

회신문서의 제목은 회신 작성자 기준으로 작성한다.

예시는 다음과 같다.

```text
# [기안답변-Claude]
# [방향점검답변-Claude]
# [검토답변-Gemini]
# [검색결과-Perplexity]
# [완료보고-Codex]
# [검토답변-ClaudeCode]
# [완료보고-ClaudeCode]
```

회신문서는 별도의 파일명을 생성하지 않는다.

사용자는 해당 회신 전문을 Drive 문서에 구분선을 삽입한 뒤 직접 붙여넣는다.

나중에 필요한 경우 답신 전문을 다시 확인할 수 있도록, 회신 전문은 요약본이 아니라 원문 전체를 보존하는 것을 원칙으로 한다.

### 3.7 AI 간 세트문서의 송신·회신 구분

AI 간 주고받는 세트문서는 송신 문서와 회신 문서를 구분한다.

송신 문서는 사용자가 다른 AI에게 전달하기 위한 요청문이다.

송신 문서는 파일명 코드블록과 하나의 문서블록 본문으로 작성한다.

송신 문서의 제목은 작성자 기준으로 작성한다.

회신 문서는 다른 AI가 요청에 답하는 결과물이다.

회신 문서는 부연설명 없이 채팅에 직접 출력하도록 요청한다.

회신 문서의 제목은 회신 작성자 기준으로 작성한다.

사용자는 회신 전문을 Drive 문서 내 구분선 아래에 붙여넣는다.

GPT가 이후 해당 회신을 해석할 때에는 다음을 분리한다.

1. 회신 원문
2. GPT 해석
3. 반영 여부
4. 사용자 결정 필요 항목
5. 보류 또는 제외 항목

### 3.8 헤딩 규격

문서본문과 AI 회신 본문은 다음 헤딩 규격을 따른다.

첫 줄의 문서 제목 또는 답변 제목은 `#`으로 작성한다.

본문의 1단계 장은 `## 1.` 형식으로 작성한다.

본문의 2단계 장은 `### 1.1` 형식으로 작성한다.

본문의 3단계 장은 필요한 경우 `#### 1.1.1` 형식으로 작성한다.

`#`은 문서 제목 또는 답변 제목에만 사용한다.

본문 장 제목에는 `#`을 사용하지 않는다.

문서 유형별 목차 순서와 내용 구성은 상황에 따라 유연하게 조정할 수 있다.

단, 제목, 헤딩 규격, 파일명 코드블록, 문서본문 문서블록, 회신규격 등 출력 틀은 유지한다.

### 3.9 헤딩 예시

이 장의 예시는 코드블록으로 작성한다.

단, 실제 문서본문 출력은 코드블록이 아니라 문서블록으로 작성한다.

발신문서 본문 헤딩 예시는 다음과 같다.

```markdown
# [작업지시-GPT]

## 1. 작업 목적

### 1.1 세부 목적

## 2. 작업 범위

### 2.1 포함 범위

### 2.2 제외 범위

## 3. 완료보고 요구사항

### 3.1 필수 보고 항목

### 3.2 금지사항 확인
```

AI 회신 본문 헤딩 예시는 다음과 같다.

```markdown
# [검토답변-Gemini]

## 1. 검토 결과 요약

### 1.1 핵심 판단

## 2. 발견된 리스크

### 2.1 구조 리스크

### 2.2 회귀 리스크

## 3. 반영 권장사항

### 3.1 즉시 반영

### 3.2 보류 가능
```

### 3.10 회신규격 요청문 필수 문구

AI에게 발신문서를 작성할 때에는 다음 취지의 회신규격을 포함한다.

```text
답변은 부연설명 없이 본문 내용만 작성해 주세요.

첫 줄은 다음 형식의 제목으로 작성해 주세요.
# [검토답변-Gemini]

본문은 다음 헤딩 규격을 따라 주세요.
## 1.
### 1.1
#### 1.1.1

확인하지 못한 항목은 추정하지 말고 “확인 불가” 또는 “추가 자료 필요”로 분리해 주세요.
```

AI 종류와 회신 유형에 따라 제목 예시는 다음처럼 바꾼다.

```text
# [기안답변-Claude]
# [방향점검답변-Claude]
# [검토답변-Gemini]
# [검색결과-Perplexity]
# [완료보고-Codex]
# [검토답변-ClaudeCode]
# [완료보고-ClaudeCode]
```

회신규격은 문서 유형에 맞게 표현을 조정할 수 있다.

단, 다음 취지는 유지한다.

1. 부연설명 없이 본문만 작성
2. 첫 줄 제목 형식 명시
3. 헤딩 규격 명시
4. 확인 불가 항목의 추정 금지
5. 확인 불가 또는 추가 자료 필요 항목 분리

## 4. 코드블록 및 터미널 명령어 출력 기준

### 4.1 코드블록 기본 원칙

터미널에 붙여넣을 명령어는 반드시 하나의 코드블록으로 제공한다.

코드, 터미널 명령, 파일 본문, 패치 본문은 서로 다른 코드블록으로 분리한다.

사용자가 그대로 복사해 실행해야 하는 내용에는 불필요한 설명문을 코드블록 안에 넣지 않는다.

### 4.2 Git 명령어 출력 기준

Git 후속작업이 필요한 경우 사용자가 별도로 요청하지 않아도 터미널용 명령어 코드블록을 제공할 수 있다.

단, Git 명령어는 사용자가 직접 실행하는 안내용으로만 제공한다.

Codex와 ClaudeCode에게 Git 저장 이력 변경 작업을 지시하지 않는다.

Git merge 안내 시 Vim 등 편집기가 뜨지 않도록 한다.

merge commit이 필요한 경우 `-m` 옵션을 포함한다.

예시는 다음과 같다.

```bash
git checkout main
git pull --ff-only
git merge --no-ff feature/example-branch -m "Merge feature/example-branch"
git push origin main
```

### 4.3 Git 상태 확인 명령어

커밋 전 확인이 필요한 경우 다음 명령어를 우선 안내한다.

```bash
git status
git branch --show-current
git log -1 --oneline
git diff --stat
```

필요한 경우 상세 diff 확인 명령을 추가한다.

```bash
git diff
```

### 4.4 Android 빌드 관련 출력 기준

Codex와 ClaudeCode에게 Android APK 또는 AAB 빌드를 지시하지 않는다.

Android 실기기 테스트가 필요한 경우 사용자가 직접 빌드·설치·실기기 테스트를 수행해야 함을 안내한다.

Android A영역, Safe Area, 상태바, 하단 시스템 제스처, 카메라홀 관련 검증은 Android 실기기 테스트를 최종 기준으로 안내한다.

## 5. 파일명 기준

### 5.1 소스 기준문서 파일명

Anima Spire 프로젝트의 소스 기준문서는 다음 4종으로 운영한다.

```text
01_AI_ROUTER.md
02_AI_WORKFLOW_RULES.md
03_AI_OUTPUT_RULES.md
IMPLEMENTATION_GUARDRAILS.md
```

`01`, `02`, `03`은 AI가 순차 참조하는 문서에만 붙인다.

`IMPLEMENTATION_GUARDRAILS.md`는 구현자 가드레일로서 별도 병렬 트랙이므로 넘버링하지 않는다.

소스 기준문서 파일명에는 버전을 붙이지 않는다.

버전은 문서 내부 첫 줄 제목과 변경 이력에만 기록한다.

예시는 다음과 같다.

```text
파일명: 02_AI_WORKFLOW_RULES.md
문서 내부 제목: # AI Workflow Rules - Anima Spire 작업판단기준_v02
변경 이력: v01, v02
```

### 5.2 Drive 저장문서 기본 파일명 구조

Drive 저장문서의 기본 파일명은 다음 구조를 따른다.

```text
YYMMDD_NN_Anima Spire {대상}_{AI명}_{문서종류}_{문서제목}_vXX
```

각 요소의 의미는 다음과 같다.

| 요소            | 의미                                                   |
| ------------- | ---------------------------------------------------- |
| `YYMMDD`      | 문서 작성일                                               |
| `NN`          | 해당 날짜의 문서 일련번호                                       |
| `Anima Spire` | 프로젝트명                                                |
| `{대상}`        | MVP 번호, 세부단계번호, 운영, 기획 등 문서 대상                       |
| `{AI명}`       | GPT, Claude, Gemini, Codex, ClaudeCode, Perplexity 등 |
| `{문서종류}`      | 기안문, 검토세트, 검색세트, 구현지시, 검토지시, 인수인계 등                  |
| `{문서제목}`      | 문서 내용을 압축한 짧은 제목                                     |
| `vXX`         | 문서 버전 또는 작업차수                                        |

### 5.3 AI명을 생략할 수 있는 문서

다음 문서는 AI명을 생략할 수 있다.

1. 전체 기획서
2. 카테고리별 기획서
3. MVP 구현로드맵
4. MVP 구현로드맵 완료반영판
5. 폐기결정
6. 인수인계
7. 테스트·이슈기록
8. 빌드·릴리즈 기록
9. 사용자가 직접 작성한 결정 기록

예시는 다음과 같다.

```text
260618_02_Anima Spire 034_UI탭구조완성및씬구조예약_MVP구현로드맵_v01
260618_03_Anima Spire 033_UI Surface및Android A영역대응_MVP구현로드맵_완료반영판_v01
260618_04_Anima Spire 인수인계_033완료및034진입전_v01
```

### 5.4 AI명을 포함하는 문서

다음 문서는 원칙적으로 AI명을 포함한다.

1. Claude 기안문
2. Claude 방향점검 요청문
3. Gemini 검토세트
4. Perplexity 검색세트
5. Codex 구현지시
6. ClaudeCode 검토지시
7. ClaudeCode 구현지시
8. GPT 정리문 또는 GPT 해석문

예시는 다음과 같다.

```text
260618_01_Anima Spire 운영_Claude_기안문_기준문서4종체계개편_v01
260618_05_Anima Spire 034A_Gemini_검토세트_UI탭구조설계검토_v01
260618_06_Anima Spire 034A_Codex_구현지시_UI탭구조MVP_v01
260618_07_Anima Spire 034A_ClaudeCode_검토지시_UI탭구조회귀검토_v01
```

### 5.5 일련번호 기준

Drive 저장문서 파일명 맨 앞에는 해당 날짜의 일련번호를 붙인다.

일련번호는 `01`, `02`, `03`처럼 두 자리로 작성한다.

파일명을 생성해야 하는 경우 AI는 먼저 Drive에서 해당 날짜의 기존 문서를 검색한다.

검색 기준은 파일명 맨 앞의 `YYMMDD`이다.

예를 들어 2026년 6월 18일에 작성하는 문서라면 `260618`로 시작하는 기존 문서를 검색한다.

검색 결과에서 가장 큰 일련번호 `NN`을 확인한 뒤, 새 문서에는 다음 번호를 사용한다.

예시는 다음과 같다.

```text
검색 결과:
260618_01_Anima Spire 운영_Claude_기안문_기준문서4종체계개편_v01
260618_02_Anima Spire 034_UI탭구조완성및씬구조예약_MVP구현로드맵_v01
260618_03_Anima Spire 인수인계_033완료및034진입전_v01

새 문서:
260618_04_Anima Spire 034A_Codex_구현지시_UI탭구조MVP_v01
```

같은 날짜에 이미 작성된 문서가 있는 경우 다음 번호를 사용한다.

Drive 검색이 가능하지만 결과가 없으면 해당 날짜의 첫 문서로 보고 `01`을 사용할 수 있다.

Drive 검색이 불가능하거나, 검색 결과가 불명확하거나, 사용자가 이미 Drive에 수동으로 작성한 문서가 있을 가능성이 있으면 AI는 일련번호를 임의로 확정하지 않는다.

이 경우 AI는 사용자에게 해당 날짜의 최신 일련번호 확인을 요청한 뒤 파일명을 제안한다.

파일명을 임시로 제안해야 하는 경우에는 `NN`을 그대로 두거나, `일련번호 확인 필요`로 표시한다.

예시는 다음과 같다.

```text
260618_NN_Anima Spire 034A_Gemini_검토세트_UI탭구조설계검토_v01
```

또는 다음과 같이 표시할 수 있다.

```text
260618_일련번호확인필요_Anima Spire 034A_Gemini_검토세트_UI탭구조설계검토_v01
```

AI는 Drive 확인 없이 해당 날짜의 일련번호를 확정값처럼 작성하지 않는다.

### 5.6 버전번호 기준

Drive 저장문서 버전은 `v01`, `v02`, `v03` 형식으로 표기한다.

소스 기준문서 내부 버전도 `v01`, `v02`, `v03` 형식으로 표기한다.

정규 기획문서 중 전체 기획서와 카테고리별 기획서는 필요에 따라 `rXXX.XX` 계열을 사용할 수 있다.

MVP 구현로드맵, 완료반영판, 인수인계, 폐기결정, AI협업문서는 원칙적으로 `vXX` 계열을 사용한다.

### 5.7 보완루프 문서 버전 기준

보완루프 이후 새로 생성되는 검토세트, 검색세트, 구현지시, 검토지시는 신규 문서로 생성한다.

보완루프 문서의 작업차수는 `v02`부터 시작한다.

재보완 시 `v03`, `v04` 형식으로 증가한다.

예시는 다음과 같다.

```text
260618_08_Anima Spire 034A_Gemini_검토세트_UI탭구조회귀위험_보완설계검토_v02
260618_09_Anima Spire 034A_Codex_구현지시_UI탭구조회귀위험_보완구현_v02
```

작업차수는 문서 수정 버전이 아니라 동일 작업단계 내 보완 차수를 의미한다.

### 5.8 폐기결정 문서 파일명

폐기결정 문서는 AI명 없이 작성한다.

기본 형식은 다음과 같다.

```text
YYMMDD_NN_Anima Spire {단계번호}_폐기결정_{문서제목}_vXX
```

예시는 다음과 같다.

```text
260618_10_Anima Spire 034A_폐기결정_UI탭구조접근방식폐기_v01
```

### 5.9 빌드 파일명 기준

Android 테스트빌드, 실기기 검증빌드, 릴리즈 후보빌드 등 빌드 산출물의 파일명은 다음 구조를 따른다.

```text
AnimaSpire_STEP{MVP단계5자리}_{작업내용}_{빌드차수2자리}
```

예시는 다음과 같다.

```text
AnimaSpire_STEP033J0_Android실기기검증및마무리_02
```

각 요소의 의미는 다음과 같다.

| 요소               | 의미                         |
| ---------------- | -------------------------- |
| `AnimaSpire`     | 프로젝트명                      |
| `STEP{MVP단계5자리}` | 빌드가 생성된 MVP 단계 식별자         |
| `{작업내용}`         | 해당 빌드의 목적 또는 작업 내용을 압축한 설명 |
| `{빌드차수2자리}`      | 해당 MVP 단계에서 생성된 빌드 차수      |

`STEP` 뒤의 MVP 단계 식별자는 5자리로 작성한다.

MVP 단계 식별자는 다음 형식을 따른다.

```text
숫자 3자리 + 영문 1자리 + 숫자 1자리
```

예시는 다음과 같다.

```text
033J0
033J3
034A0
```

`033J0`은 `033J` 단계에서 생성된 기본 빌드 또는 `033J-0`에 해당하는 빌드를 의미한다.

`033J3`은 `033J-3` 단계에서 생성된 빌드를 의미한다.

작업내용은 해당 빌드의 목적을 짧게 설명한다.

작업내용에는 필요한 경우 영문과 한글을 함께 사용할 수 있다.

빌드차수는 해당 MVP 단계에서 생성된 빌드 순서를 의미한다.

첫 번째 빌드는 `_01`, 두 번째 빌드는 `_02`, 세 번째 빌드는 `_03` 형식으로 작성한다.

예시는 다음과 같다.

```text
AnimaSpire_STEP033J0_Android실기기검증및마무리_01
AnimaSpire_STEP033J0_Android실기기검증및마무리_02
AnimaSpire_STEP033J3_UI보완빌드_01
AnimaSpire_STEP034A0_탭구조초기검증_01
```

빌드 파일명은 빌드 산출물 자체의 이름에 적용한다.

빌드·릴리즈 기록 문서의 파일명은 Drive 저장문서 파일명 기준을 따른다.

Codex와 ClaudeCode는 Android APK 또는 AAB 빌드를 수행하지 않는다.

## 6. Drive 폴더 저장 기준

### 6.1 기본 원칙

Drive는 기획문서, MVP 로드맵, AI협업문서, 인수인계, 테스트·이슈기록, 빌드·릴리즈 자료, 운영기록, 임시문서, 아카이브 문서를 보관하는 공간으로 운영한다.

소스 기준문서 4종은 GitHub 소스에 저장한다.

Drive에는 기준문서 개편 기안문, 기준문서 검토문, 기준문서 최종검토 결과 등 협업 산출물을 저장할 수 있다.

### 6.3 단계별 저장 기준

Drive 저장 위치는 문서 유형만으로 판단하지 않고, 해당 문서가 발생한 작업 단계와 목적을 기준으로 판단한다.

검토세트와 검색세트는 기획단계와 작업단계 양쪽에서 모두 발생할 수 있으므로, 문서 유형만으로 저장 위치를 고정하지 않는다.

같은 Gemini 검토세트라도 구현 전 기획, 방향, 구조, 범위, 로드맵 판단을 위한 문서이면 `03_AI협업문서/03-02_기획단계협업`에 저장한다.

같은 Gemini 검토세트라도 특정 MVP 또는 세부 작업단계의 구현, 보완, 코드검토, 완료보고, 테스트 대응과 연결된 문서이면 `03_AI협업문서/03-03_작업단계협업`에 저장한다.

외부 정책, 공식문서, 기술 이슈 검색이 기획 판단을 위한 것이면 `03_AI협업문서/03-02_기획단계협업`에 저장한다.

### 6.4 세트문서 저장 방식

세트문서는 발신문서와 회신문서를 하나의 Drive 문서 안에서 관리한다.

GPT는 발신문서를 문서블록으로 작성한다.

사용자는 발신문서를 Drive 문서로 저장한 뒤 다른 AI에게 전달한다.

다른 AI의 회신은 부연설명 없이 채팅에 직접 출력하도록 요청한다.

사용자는 Drive 문서 안에 구분선을 삽입한 뒤 회신 전문을 붙여넣는다.

권장 구분선은 다음과 같다.

```text
---

# [회신 원문]
```

필요한 경우 GPT는 회신 이후 다음 구분을 추가로 작성할 수 있다.

1. GPT 해석
2. 반영 여부
3. 사용자 결정 필요 항목
4. 보류 또는 제외 항목

### 6.5 저장 안내 방식

파일명을 생성하는 Drive 저장문서 작성 요청의 경우 AI는 다음 정보를 안내할 수 있다.

1. 추천 파일명
2. 권장 저장 폴더
3. 문서 성격
4. 후속 검토 필요 여부
5. 사용자가 다음에 해야 할 작업

단, 위 항목의 출력 순서는 고정하지 않는다.

사용자의 요청 맥락에 따라 필요한 안내만 작성할 수 있다.

파일명은 반드시 코드블록으로 작성한다.

Drive에 저장할 문서본문은 반드시 하나의 마크다운 렌더링 문서블록으로 작성한다.

저장 위치 안내, 문서작성 모드 안내, 후속 검토 필요 여부, 작업 설명은 문서본문 밖 일반 채팅 영역에 작성할 수 있다.

문서작성 모드 안내 문구는 문서본문 안에 포함하지 않는다.

예시는 다음과 같다.

```text
문서작성 모드로 처리합니다.
```

저장 안내 예시는 다음과 같다.

```text
권장 저장 위치: 03_AI협업문서/03-02_기획단계협업
```

다만 저장 위치 안내가 문서본문보다 반드시 앞서야 하는 것은 아니다.

사용자가 파일명과 본문만 요청한 경우에는 저장 위치 안내를 생략할 수 있다.

## 7. 문서 유형별 출력 기준

### 7.1 검토세트

검토세트는 GPT 또는 Claude가 다른 AI에게 검토를 요청하기 위해 작성하는 발신문서와, 해당 AI가 작성하는 회신으로 구성된다.

검토세트는 다음 상황에서 사용한다.

1. 기획안, 구현로드맵, 기준문서, 구현 방향에 대해 다른 AI의 검토가 필요한 경우
2. Gemini에게 기술 검토를 요청하는 경우
3. Claude에게 기획·구조·방향 검토를 요청하는 경우
4. ClaudeCode에게 코드검토를 요청하는 경우

검토요청문은 사용자가 명시적으로 문서작성을 요청한 경우에만 파일명 생성 대상 문서로 작성한다.

검토요청문의 파일명은 코드블록으로 작성한다.

검토요청문의 본문은 하나의 마크다운 렌더링 문서블록으로 작성한다.

검토요청문의 제목은 작성자 기준으로 작성한다.

예시는 다음과 같다.

```text
# [검토요청-GPT]
```

검토답변의 제목은 회신 작성자 기준으로 작성하도록 요청한다.

예시는 다음과 같다.

```text
# [검토답변-Gemini]
# [검토답변-Claude]
# [검토답변-ClaudeCode]
```

검토요청문에는 다음 항목을 포함할 수 있다.

1. 검토 목적
2. 현재 상황
3. 검토 대상
4. 검토 질문
5. 반영 가능 범위
6. 제외 범위
7. 확인해야 할 기준문서
8. 회신 출력규격

검토요청문의 목차 순서와 내용 구성은 상황에 따라 유연하게 조정할 수 있다.

단, 다음 출력 틀은 유지한다.

1. 파일명 코드블록
2. 하나의 문서블록 본문
3. 작성자 기준 제목
4. 헤딩 규격
5. 회신 출력규격
6. 확인 불가 항목 분리 요청

Gemini 기술검토 요청문에 구현 구조, 회귀 위험, SaveData, Scene, UI, Android, Combat, Equipment, Legacy code 관련 검토가 포함되는 경우에는 `IMPLEMENTATION_GUARDRAILS.md`의 관련 장을 함께 대조하도록 요청한다.

이 경우 구현 가드레일 전체를 무조건 붙여넣지 않고, 검토 대상과 직접 관련 있는 항목만 요약하여 포함한다.

검토요청문에는 다음 취지의 회신규격을 포함한다.

```text
답변은 부연설명 없이 본문 내용만 작성해 주세요.

첫 줄은 다음 형식의 제목으로 작성해 주세요.
# [검토답변-Gemini]

본문은 다음 헤딩 규격을 따라 주세요.
## 1.
### 1.1
#### 1.1.1

확인하지 못한 항목은 추정하지 말고 “확인 불가” 또는 “추가 자료 필요”로 분리해 주세요.
```

수신 AI에 따라 제목 예시는 조정한다.

검토답변은 파일명을 생성하지 않는다.

### 8.2 세트문서

세트문서는 요청문과 회신 원문, 그리고 후속 해석 또는 반영 판단을 함께 관리하는 문서다.

세트문서는 사용자가 명시적으로 문서작성을 요청한 경우에만 파일명 생성 대상 문서로 작성한다.

2. Perplexity 검색세트
3. Claude 방향점검 세트
4. Claude 기안문 검토세트
5. Codex 작업지시 및 완료보고 세트
6. ClaudeCode 검토지시 및 검토답변 세트

세트문서의 최초 발신문서는 파일명 코드블록과 하나의 마크다운 렌더링 문서블록으로 작성한다.

발신문서의 제목은 작성자 기준으로 작성한다.

```text

# [검색요청-GPT]
# [기안요청-GPT]
# [작업지시-GPT]
# [검토지시-GPT]
```

회신 원문의 제목은 회신 작성자 기준으로 작성하도록 요청한다.

예시는 다음과 같다.

```text
# [검토답변-Gemini]
# [검색결과-Perplexity]
# [기안답변-Claude]
# [완료보고-Codex]
# [검토답변-ClaudeCode]
```

세트문서의 기본 관리 구조는 다음과 같다.

```markdown
# [검토요청-GPT]

## 1. 요청 목적

## 2. 요청 내용

## 3. 회신 출력규격

---

# [검토답변-Gemini]

## 1. 검토 결과 요약

## 2. 발견된 문제

## 3. 반영 권장사항

---

# [GPT 해석]

## 1. 핵심 판단

## 2. 반영 여부

## 3. 사용자 결정 필요 항목

## 4. 보류 또는 제외 항목
```

세트문서의 목차 순서와 내용 구성은 상황에 따라 유연하게 조정할 수 있다.

단, 다음 출력 틀은 유지한다.

1. 최초 발신문서의 파일명 코드블록
2. 최초 발신문서의 하나의 문서블록 본문
3. 작성자 기준 발신문서 제목
4. 회신 작성자 기준 회신문서 제목
5. 헤딩 규격
6. 회신 출력규격
7. 확인 불가 항목 분리 요청
8. 회신 원문 보존 구조

회신 원문은 요약본으로 대체하지 않는다.

사용자는 회신 전문을 원 세트문서의 구분선 아래에 직접 붙여넣는다.

GPT 또는 Claude가 회신을 해석할 때에는 회신 원문과 해석을 분리한다.

회신 해석에는 다음 항목을 포함할 수 있다.

1. 핵심 판단
2. 반영 가능 항목
3. 반영 보류 항목
4. 반영 제외 항목
5. 사용자 결정 필요 항목
6. 추가 확인 필요 항목

다른 AI에게 전달할 내용 작성 요청이라도, 사용자가 명시적으로 문서 작성 또는 `[문서]` 트리거를 요청하지 않은 경우에는 세트문서로 자동 작성하지 않는다.

### 8.3 지시문검토 문서화 기준

지시문검토는 정식 Drive 저장문서로 만들지 않는 것을 원칙으로 한다.

따라서 파일명 생성 규격을 적용하지 않는다.

지시문검토는 기본적으로 채팅 내 검토로 처리한다.

다만 지시문검토 결과를 반영해 추가 지시문구를 작성하는 경우, 해당 추가 지시문구는 후속 작업지시문 보완 또는 보완루프 문서에 반영할 수 있다.

## 11. 기준문서 개정 출력 기준

### 11.1 기준문서 초안 작성 방식

기준문서 초안은 문서별로 작성한다.

한 응답에서 4개 기준문서 전체를 모두 완성하려고 하지 않는다.

기준문서는 원칙적으로 다음 순서로 작성한다.

1. `02_AI_WORKFLOW_RULES.md`
2. `03_AI_OUTPUT_RULES.md`
3. `IMPLEMENTATION_GUARDRAILS.md`
4. `01_AI_ROUTER.md`

단, 사용자가 요청하면 임시 라우터 초안을 먼저 작성할 수 있다.

### 11.2 사용자 검토 방식

기준문서 초안을 작성한 뒤 AI는 다음을 사용자에게 확인한다.

1. 수정이 필요한 항목
2. 사용자 결정이 필요한 항목
3. 삭제 또는 축약할 항목
4. 다른 문서로 이관할 항목
5. Claude 최종검토가 필요한 시점

### 11.3 최종본 작성 방식

최종본에서는 초안 작성 경위, 기존 문서체계의 문제점, 개편 배경 설명을 본문에 길게 쓰지 않는다.

최종본은 앞으로 적용할 기준만 명확히 적는다.

개편 경위는 변경 이력에 간단히 기록한다.

### 11.4 기준문서 내부 버전 표기

소스 기준문서의 파일명에는 버전을 붙이지 않는다.

소스 기준문서의 파일명은 다음처럼 유지한다.

```text
01_AI_ROUTER.md
02_AI_WORKFLOW_RULES.md
03_AI_OUTPUT_RULES.md
IMPLEMENTATION_GUARDRAILS.md
```

문서 내부 첫 줄 제목에는 `_vXX` 형식의 버전을 붙인다.

변경 이력은 `v01`, `v02`, `v03` 형식으로 작성한다.

예시는 다음과 같다.

```markdown
# AI Output Rules - Anima Spire 출력기준_v03

## 12. 변경 이력

v01

* 최초 작성.

v02

* 사용자 결정사항 반영.

v03

* 문서작성 요청 판단 기준과 출력기준 적용 범위 정리.
```

동일한 기준문서 개정 작업에서 함께 수정되는 기준문서들은 같은 버전 번호를 사용한다.

일부 문서만 변경되는 경우에도, 해당 개정 세션의 기준문서 묶음에서 사용자에게 확정받은 버전 번호를 따른다.

버전 번호를 올릴지 여부가 애매한 경우 AI가 임의로 판단하지 않고 사용자에게 확인한다.

### 11.5 소스 기준문서 부분 교체 출력 기준

소스 기준문서 md 4종을 개정할 때에는 변경되는 부분만 문서블록으로 출력한다.

이 절의 적용 대상은 다음 4종이다.

1. `01_AI_ROUTER.md`
2. `02_AI_WORKFLOW_RULES.md`
3. `03_AI_OUTPUT_RULES.md`
4. `IMPLEMENTATION_GUARDRAILS.md`

소스 기준문서 개정 출력 시 AI는 원칙적으로 문서 전체를 재생성하지 않는다.

변경되는 장·절이 포함된 가장 작은 헤딩 단위 전체를 교체본문으로 작성한다.

가장 작은 헤딩 단위란 변경 내용이 포함된 `#`, `##`, `###`, `####` 중 실제 교체가 가능한 최소 범위의 전체 블록을 말한다.

예시는 다음과 같다.

1. `### 7.1` 일부 문장이 변경되는 경우 `### 7.1` 전체를 출력한다.
2. `## 3` 전체 구조가 변경되는 경우 `## 3` 전체를 출력한다.
3. `### 7.1`과 `### 7.3`이 변경되고 `### 7.2`가 변경되지 않는 경우 `### 7.1`과 `### 7.3`을 각각 별도 문서블록으로 출력한다.
4. `### 7.1`부터 `### 7.5`까지 연속으로 변경되는 경우 하나의 문서블록으로 묶어 출력할 수 있다.
5. 신설 장으로 인해 뒤 장 번호가 밀리는 경우, 실제 본문 변경이 없는 밀린 장은 장 제목 변경 필요 여부만 언급하고 본문 전체를 재출력하지 않는다.

연속된 변경 구간은 하나의 문서블록으로 묶을 수 있다.

연속되지 않은 변경 구간은 서로 다른 문서블록으로 분리한다.

중간에 변경되지 않는 장·절이 있는 경우, 앞뒤 변경 구간을 하나의 문서블록으로 합치지 않는다.

문서블록이 많거나 본문이 길어 한 번에 출력하기 어려운 경우에는 여러 단계로 나누어 출력한다.

여러 단계로 나누어 출력하는 경우 AI는 먼저 다음을 간단히 브리핑한다.

1. 변경되는 장·절 수
2. 생성해야 할 문서블록 수
3. 단계별 출력 범위

그 다음 단계별로 교체본문을 출력한다.

소스 기준문서 개정 시 파일 첫 줄 제목의 내부 버전 표기만 변경되는 경우에는 첫 줄 제목만 별도 문서블록으로 출력한다.

변경 이력은 실제 변경이 있는 경우 해당 `## 변경 이력` 장 전체를 교체본문으로 출력한다.

이 기준은 사용자가 소스 기준문서 전체 재작성 또는 전체 교체를 명시적으로 요청한 경우에는 예외로 할 수 있다.

v03

* 현재 요청이 문서작성 요청인지 여부는 `02_AI_WORKFLOW_RULES.md`의 문서작성 요청 판단 기준을 따르고, 이 문서는 문서작성으로 판단된 이후의 출력 형식을 정하는 문서로 역할을 정리.
* 명시적 문서작성 요청 또는 `[문서]` 트리거가 없는 일반 검토, 설명, 요약, 정리, 전달문 작성 요청에는 저장문서 출력규격을 자동 적용하지 않는 기준 추가.
* 다른 AI에게 전달할 내용 작성 요청이라도 명시적 문서작성 요청이 없으면 발신문서 출력규격을 자동 적용하지 않는 기준 추가.
